# RAG你是怎么评测的？

在面试中如果被问 **“RAG 是怎么评测的？”**，比较成熟的回答通常是：

> **RAG 评测需要分层进行，因为它是一个 pipeline，而不是单一模型。**
> 
> 
> 一般会从 **检索层（Retrieval）→ 上下文层（Context）→ 生成层（Generation）→ 系统层（End-to-End）** 四个层面评测。
> 

---

## 1 检索层评测（Retrieval Evaluation）

首先评测 **检索是否能找到正确知识**。

常见指标：

- **Recall@K（召回率）**
    
    正确文档是否出现在前 K 条结果中
    
- **MRR（Mean Reciprocal Rank，平均倒数排名）**
- **nDCG（Normalized Discounted Cumulative Gain，归一化折损累计增益）**

例如：

用户问：

```
退货运费谁承担？
```

如果正确文档没有被检索出来，即使模型很强也无法回答正确。

---

## 2 上下文质量评测（Context Evaluation）

即使检索到了正确文档，也需要评测：

```
检索结果是否真的对问题有帮助
```

常见指标：

- **Context Precision（上下文相关性）**
- **Context Recall（上下文覆盖度）**

问题可能包括：

- chunk 切分不合理
- 文档噪声过多
- 上下文长度过长

---

## 3 生成层评测（Generation Evaluation）

第三层评测模型回答是否正确。

常见指标：

- **Correctness（正确性）**
- **Relevance（相关性）**
- **Completeness（完整性）**

同时还要关注：

- **Hallucination Rate（幻觉率）**

---

## 4 知识支撑评测（Groundedness）

RAG 系统特别重要的一点是：

```
回答是否基于知识库
```

常见指标：

- **Faithfulness（事实一致性）**
- **Groundedness（知识支撑度）**
- **Citation Accuracy（引用准确率）**

例如：

模型回答：

```
根据平台政策，签收7天内可以退货
```

需要确认：

- 是否来自知识库
- 是否真实存在该政策

---

## 5 系统级评测（End-to-End）

最后还需要评测 **整体用户体验**：

常见指标：

- **Task Success Rate（任务成功率）**
- **Latency（响应时间）**
- **Cost（推理成本）**
- **User Satisfaction（用户满意度）**

---

## 总结

> RAG 评测通常需要分层进行：首先评测检索层，例如 Recall@K 或 MRR，判断是否能够找到正确知识；其次评测上下文质量，例如 Context Precision；然后评测生成回答的正确性和幻觉率；最后还需要评测回答是否真正基于知识库，例如 Faithfulness 或 Groundedness。除此之外，还需要从系统层面评估任务成功率、延迟和成本等指标。
>

# **如果 Retrieval Recall 很高，但最终回答仍然错误，你怎么定位问题？**

如果 **Retrieval Recall 很高，但最终回答仍然错误**，说明 **问题大概率不在“是否检索到正确文档”，而是在 RAG pipeline 的后续环节**。

我一般会按 **四个层级逐步定位：上下文质量 → Prompt → 模型生成 → 系统流程**。

---

## 1 检查上下文质量（Context Quality）

虽然 **Recall@K 很高**，但不代表检索结果真的适合回答问题。

可能的问题：

- 正确文档被 **淹没在大量无关文档中**
- chunk 切分不合理
- 上下文过长

例如：

```
Top5 文档中有1个正确文档
但其他4个文档无关
```

模型可能会：

```
被噪声干扰
```

需要检查：

- **Context Precision**
- 文档排序（rerank）
- chunk strategy

---

## 2 检查 Prompt 设计

如果 Prompt 没有明确约束模型：

```
必须基于上下文回答
```

模型可能会：

- 使用自己的知识
- 编造答案

例如 Prompt 没有要求：

```
If the answer is not in the context, say you don't know
```

这会导致：

```
hallucination
```

---

## 3 检查模型生成能力

即使上下文正确，模型也可能：

- 理解错误
- 推理错误

例如：

上下文：

```
签收7天内可退货
```

模型回答：

```
签收30天内可退货
```

这属于：

```
generation error
```

---

## 4 检查系统流程

有时候问题来自 **pipeline 设计**：

例如：

```
retrieval → context truncation → generation
```

可能发生：

- 正确文档被截断
- 上下文顺序不合理
- token 限制导致信息丢失

---

## 面试总结

> 如果 Retrieval Recall 很高但最终回答仍然错误，我通常会从 RAG pipeline 的后续环节定位问题。首先检查上下文质量，例如是否存在噪声文档或 chunk 切分不合理；其次检查 prompt 是否约束模型基于上下文回答；然后分析模型生成是否出现理解或推理错误；最后检查系统流程，例如上下文截断或排序问题。通过分层分析可以定位问题是在检索、上下文还是生成阶段。
> 

---

# **RAG 的 Recall@K 应该越高越好吗**

一个比较成熟的回答通常是：

> **Recall@K 不是越高越好，因为过高的 Recall 往往意味着引入更多无关文档，反而会降低最终回答质量。**
> 

关键要说明 **Recall 和上下文质量之间的 trade-off**。

---

## 1 Recall@K 的作用

**Recall@K** 衡量的是：

```
正确文档是否出现在前 K 个检索结果中
```

例如：

用户问题：

```
退货运费谁承担？
```

如果正确文档在 **Top5** 结果中：

```
Recall@5 = 1
```

这说明检索系统**找到了正确知识**。

---

## 2 Recall 太高可能带来噪声

如果为了提高 Recall，把 K 设置得很大：

```
Top20 / Top50
```

虽然 Recall 会提高，但也会带来问题：

- 无关文档增加
- 上下文噪声变多
- 模型更容易被干扰

例如：

```
Top10 文档
只有1个相关
9个无关
```

模型可能：

```
理解错误
```

---

## 3 RAG 更重要的是 Precision

在实际 RAG 系统中，往往需要平衡：

- **Recall@K（召回）**
- **Context Precision（上下文相关性）**

因为模型输入上下文是有限的：

```
token window
```

如果噪声太多，反而会降低生成质量。

---

## 4 实际工程中的做法

很多系统会通过：

**两阶段检索**

```
retrieval → rerank → generation
```

步骤：

1️⃣ 先高 Recall 检索

2️⃣ 再用 reranker 提高 Precision

这样既保证：

```
正确文档被召回
```

又减少：

```
噪声文档
```

---

## 面试总结

> Recall@K 并不是越高越好。虽然高 Recall 可以提高正确文档被检索到的概率，但如果引入过多无关文档，会增加上下文噪声，从而影响模型生成质量。因此在 RAG 系统中通常需要在 Recall 和 Context Precision 之间进行平衡，并通过 rerank 等方法提高检索结果的相关性。
>

# **你觉得 RAG 系统中最重要的指标是什么？**

在面试里，如果被问 **“RAG 系统最重要的指标是什么？”**，比较成熟的回答通常不是只说一个指标，而是说明 **RAG 的核心目标是什么，然后再给出关键指标**。

一个比较专业的回答一般是：

> **RAG 的核心目标是让模型基于知识库给出正确答案，因此最重要的指标其实是回答的正确性和是否基于知识库（groundedness）。**
> 

我通常会这样展开。

---

## 1 最核心指标：Answer Correctness（回答正确性）

RAG 系统最终目的是：

```
回答用户问题
```

因此最重要的指标仍然是：

**Answer Correctness（回答正确性）**

例如客服场景：

用户问：

```
退货运费谁承担？
```

系统回答必须：

- 符合知识库政策
- 没有错误信息

如果回答错误，即使：

- Recall 很高
- 检索很好

系统仍然是失败的。

---

## 2 第二重要指标：Groundedness / Faithfulness

RAG 的核心价值在于：

```
减少幻觉
```

因此需要评测：

**回答是否基于知识库**

常见指标：

- **Groundedness（知识支撑度）**
- **Faithfulness（事实一致性）**

例如：

如果模型回答：

```
退货30天内可以退款
```

但知识库没有这个政策，就属于：

```
hallucination
```

---

## 3 上游关键指标：Retrieval Recall

虽然不是最终指标，但检索指标也很重要：

- **Recall@K**

因为如果正确知识没有被检索到：

```
模型不可能回答正确
```

---

## 4 系统级指标：Task Success Rate

在实际应用中，还需要关注：

- **Task Success Rate**
- **Latency**
- **Cost**

例如客服系统：

```
是否解决用户问题
```

---

## 面试总结

> 在 RAG 系统中，最重要的指标通常是回答正确性，因为系统最终目标是正确回答用户问题。同时还需要关注回答是否基于知识库，例如 Groundedness 或 Faithfulness，以避免幻觉。检索指标例如 Recall@K 也很重要，因为如果正确知识没有被检索到，模型就无法回答正确。因此 RAG 评测通常需要同时关注生成质量和检索效果。
>

# **RAG 和 Fine-tuning，你会怎么选？**

这个问题其实是在考察 **系统设计能力**。一个成熟的回答通常不会简单说选哪一个，而是说明 **RAG 和 Fine-tuning 各自解决什么问题，以及在什么场景下使用**。

核心观点通常是：

> **RAG 和 Fine-tuning 不是替代关系，而是解决不同问题的两种技术。**
> 

---

## 1 RAG 适合解决“知识问题”

RAG（Retrieval-Augmented Generation）的核心作用是：

```
让模型使用外部知识
```

适合的场景包括：

- 知识频繁更新
- 企业内部知识
- 文档问答

例如客服场景：

- 退款政策
- 物流规则
- 产品说明

这些知识：

```
经常变化
```

如果用 fine-tuning，每次更新都需要重新训练模型。

---

## 2 Fine-tuning 适合解决“能力问题”

Fine-tuning 更适合优化：

```
模型行为
```

例如：

- 输出格式
- 推理能力
- 对话风格

例如客服系统：

- 更礼貌的回复
- 更结构化的回答

这些不是知识问题，而是：

```
行为模式
```

---

## 3 实际系统通常是组合使用

在很多实际 AI 应用中：

```
Fine-tuning + RAG
```

例如：

- Fine-tuning 提升模型能力
- RAG 提供最新知识

这样可以同时获得：

- 稳定能力
- 最新信息

---

## 面试总结

> RAG 和 Fine-tuning 解决的是不同类型的问题。RAG主要用于引入外部知识，适合知识频繁更新的场景，例如企业知识库或客服政策；而 Fine-tuning 更适合优化模型行为，例如输出格式或对话风格。在实际系统中通常会结合两者使用，通过 Fine-tuning 提升模型能力，同时使用 RAG 提供最新知识。
> 

---

# **什么时候 RAG 会失败？**

这是一个在 **RAG / AI评测岗位面试**中非常常见的问题。比较成熟的回答通常是：

> **RAG 会在知识问题、检索问题、理解问题或任务类型不适合时失败。**
> 

也就是 **RAG pipeline 的不同环节都可能成为失败点**。我一般会从 **四类典型场景**来解释。

---

## 1 知识库本身不完整或错误

RAG 的前提是：

```
知识库中存在正确答案
```

如果知识库：

- 没有覆盖用户问题
- 内容过时
- 信息错误

即使检索和模型都正常，系统也无法回答正确。

例如客服场景：

用户问：

```
新的退款政策是什么？
```

但知识库还没有更新。

---

## 2 检索系统没有找到正确文档

即使知识库有正确答案，也可能因为：

- embedding 不匹配
- query 表达不清
- 文档 chunk 不合理

导致：

```
正确文档没有被检索到
```

这时模型就只能：

```
猜答案
```

容易产生 **hallucination**。

---

## 3 上下文噪声过多

有时候问题不是 **检索不到**，而是 **检索太多无关内容**。

例如：

```
Top10 文档
只有1个相关
9个无关
```

模型可能：

- 被噪声干扰
- 理解错误

因此 RAG 不只是 Recall，还需要 **Context Precision**。

---

## 4 任务本身不适合 RAG

RAG 主要解决的是：

```
知识检索问题
```

但如果任务是：

- 数学推理
- 复杂逻辑推理
- 代码生成

这些问题：

```
不依赖外部知识
```

RAG 的效果就非常有限。

---

## 面试总结

> RAG 主要在四种情况下可能失败：第一是知识库本身不完整或过时；第二是检索系统没有找到正确文档；第三是上下文噪声过多导致模型理解错误；第四是任务本身并不依赖外部知识，例如复杂推理任务。因此在实际系统中需要同时优化知识库质量、检索效果以及上下文设计。
> 

---

# **如果知识库很大，你怎么设计 chunk strategy？**

这是一个 **RAG 工程 + 评测结合的问题**。回答时最好体现两个思路：

1️⃣ chunk 设计是为了 **提高检索质量**

2️⃣ chunk strategy 需要 **通过评测不断优化**

一个比较完整的回答通常包括 **设计原则 + 常见策略 + 如何评测优化**。

---

## 1 设计原则：保证语义完整

chunk 切分最核心的原则是：

```
一个 chunk 应该包含完整语义
```

如果 chunk 太小：

- 信息不完整
- 模型无法理解上下文

例如：

错误切分：

```
chunk1: 退货政策...
chunk2: 7天内...
```

模型可能无法理解：

```
7天内可以退货
```

---

## 2 控制 chunk 大小

chunk 过大或过小都会影响检索。

一般经验值：

- **200–500 tokens**

如果 chunk 太大：

```
检索精度下降
```

如果 chunk 太小：

```
语义丢失
```

---

## 3 使用 overlap 保留上下文

为了避免语义被截断，通常会使用：

**chunk overlap**

例如：

```
chunk1: tokens 1–300
chunk2: tokens 250–550
```

这样可以避免：

```
关键信息被切断
```

---

## 4 按语义结构切分

相比固定 token 切分，更好的方式是：

**结构化切分**

例如：

- 按段落
- 按标题
- 按 FAQ 问答

这样每个 chunk 更完整。

---

## 5 用评测来优化 chunk strategy

chunk strategy 不应该只凭经验，需要通过评测优化，例如：

观察指标：

- **Recall@K**
- **Context Precision**
- **Answer Correctness**

例如：

```
不同 chunk size → 对比 Recall@K
```

选择效果最好的方案。

---

## 面试总结

> 在知识库较大的情况下，chunk strategy 的设计通常遵循三个原则：首先保证每个 chunk 具有完整语义，其次控制 chunk 大小，例如 200–500 tokens，并通过 overlap 避免语义被截断；此外尽量按语义结构切分，例如按段落或标题。最后需要通过检索指标如 Recall@K 或回答正确率等评测指标不断优化 chunk 策略。
> 

---

# **chunk size 应该越大越好吗？**

不是的。**chunk size 并不是越大越好**。在 RAG 系统中，chunk size 需要在 **语义完整性和检索精度之间取得平衡**。如果 chunk 太大或太小都会带来问题。

---

## 1 chunk 太大会降低检索精度

如果 chunk 很大，例如：

```
1000+ tokens
```

可能包含：

- 多个主题
- 无关信息

这样在向量检索时：

```
embedding 表达会变得模糊
```

例如一个 chunk 同时包含：

- 退款政策
- 物流说明

用户问：

```
退货运费谁承担？
```

检索可能不够精准。

---

## 2 chunk 太小会丢失语义

如果 chunk 太小，例如：

```
50 tokens
```

可能导致：

- 语义不完整
- 关键上下文丢失

例如：

```
chunk1: 退货政策...
chunk2: 7天内...
```

模型无法理解完整规则。

---

## 3 实际工程中的折中

在实际系统中，chunk size 通常会控制在：

```
200–500 tokens
```

同时配合：

- **overlap（重叠）**
- **rerank（重排序）**

保证：

- Recall
- Context precision

之间的平衡。

---

## 4 最终需要通过评测确定

chunk size 不应该只凭经验确定，而需要通过评测优化，例如：

对比不同配置：

| chunk size | Recall@K | Answer Accuracy |
| --- | --- | --- |
| 200 | 0.82 | 0.78 |
| 400 | 0.88 | 0.84 |
| 800 | 0.89 | 0.79 |

选择：

```
整体效果最好的方案
```

---

## 面试总结

> chunk size 并不是越大越好。chunk 太大会降低检索精度，因为一个 chunk 可能包含多个主题；而 chunk 太小又可能导致语义不完整。因此在实际系统中通常会在语义完整性和检索精度之间取得平衡，例如使用 200–500 tokens 的 chunk，并通过 overlap 和评测指标不断优化。
>

# **RAG 系统里最难评测的部分是什么？**

如果在 **RAG / AI评测岗位面试**中被问这个问题，一个比较成熟的回答通常是：

> **RAG 系统里最难评测的部分是回答是否真正“基于知识库”（groundedness / faithfulness）。**
> 

因为 RAG 的目标不仅是回答正确，还要 **确保答案来源于知识库，而不是模型自身的幻觉或已有知识**。

---

## 1 为什么 Groundedness 最难评测

RAG 的核心价值是：

```
让模型基于外部知识回答
```

但现实中会出现一种情况：

```
回答看起来是正确的
但其实没有来自知识库
```

例如：

知识库内容：

```
退货政策：签收7天内可退货
```

模型回答：

```
根据平台政策，签收7天内可以退货
```

这个答案：

- **正确**
- 但可能 **不是从检索文档推理出来**

而是模型自己知道。

---

## 2 为什么这会带来问题

如果模型：

```
不用知识库也能回答
```

会带来风险：

- 当知识更新时
- 模型仍然给出旧答案

例如：

```
政策从7天改为3天
```

模型仍然回答：

```
7天
```

---

## 3 实际评测方式

为了评测 groundedness，通常会使用：

**方法1：引用评测**

检查回答是否引用正确文档。

---

**方法2：LLM-as-judge**

让 judge 模型判断：

```
答案是否被上下文支持
```

---

**方法3：对抗测试**

例如：

```
context 中没有答案
```

模型是否会：

```
拒答
```

---

## 面试总结

> 在 RAG 系统中，最难评测的部分通常是回答是否真正基于知识库，也就是 groundedness 或 faithfulness。因为一个回答即使是正确的，也可能并不是从检索到的知识中推理出来，而是模型自身的知识。如果模型没有真正使用知识库，就可能在知识更新时产生错误。因此在评测中通常需要结合引用检查、LLM-as-judge 以及对抗测试来判断回答是否被上下文支持。
> 

---

# **如果模型回答正确，但不是来自知识库，你会认为 RAG 成功了吗？**

这是一个**RAG 评测里非常典型的“陷阱题”**。一个比较成熟的回答通常是：

> **从用户体验角度看，这次回答是成功的；但从 RAG 系统设计角度看，这并不算完全成功。**
> 

关键是要说明 **RAG 的目标不仅是回答正确，还要确保答案来自知识库**。

---

## 1 从用户角度：算成功

如果用户问：

```
退货政策是什么？
```

模型回答：

```
签收7天内可以退货
```

而这个答案是正确的，那么从用户体验来看：

```
用户问题被解决
```

这次交互是成功的。

---

## 2 从 RAG 系统角度：不算完全成功

RAG 系统的核心目标是：

```
让模型依赖知识库
```

如果模型回答正确，但 **没有使用检索到的知识**，说明：

- RAG pipeline 没发挥作用
- 模型依赖的是 **预训练知识**

这会带来风险：

```
知识更新时模型仍然回答旧信息
```

例如：

- 政策从 **7天 → 3天**

模型仍然回答 **7天**。

---

## 3 在评测体系中的处理方式

因此在评测中通常会区分两个指标：

| 指标 | 含义 |
| --- | --- |
| Answer Correctness | 回答是否正确 |
| Groundedness / Faithfulness | 回答是否基于知识库 |

只有同时满足：

```
正确 + 基于知识库
```

才算 **RAG 完整成功**。

---

## 面试总结

> 如果模型回答正确但没有使用知识库，从用户角度看这次回答是成功的，但从 RAG 系统设计角度看并不算完全成功。因为 RAG 的目标不仅是回答正确，还要确保答案来源于知识库，以保证知识更新后系统能够保持一致性。因此在评测体系中通常需要同时评估回答正确性和 groundedness。
> 

---

# **你怎么判断模型有没有真的使用知识库？**

这是 **RAG 评测中最核心的问题之一**。一个成熟的回答通常不会只说一种方法，而是说明 **可以从多个角度判断模型是否真正使用了知识库**。常见做法一般分为 **三类：引用检查、上下文一致性评测、对抗测试**。

---

## 1 引用检查（Citation Check）

最直接的方法是要求模型：

```
在回答中引用知识来源
```

例如：

用户问：

```
退货政策是什么？
```

模型回答：

```
根据《售后政策》第3条，签收7天内可以退货。
```

然后检查：

- 引用的文档是否真实存在
- 引用内容是否支持回答

常见指标：

- **Citation Accuracy（引用准确率）**

---

## 2 上下文一致性评测（Groundedness）

第二种方法是判断：

```
回答是否被检索到的上下文支持
```

通常会用：

**LLM-as-judge**

让评测模型判断：

- 回答是否来自 context
- 是否出现 hallucination

常见指标：

- **Faithfulness（事实一致性）**
- **Groundedness（知识支撑度）**

---

## 3 对抗测试（Ablation / Negative Test）

第三种方法是做 **对抗实验**。

例如：

### 方法1：移除上下文

```
question + 无context
```

如果模型仍然给出同样答案：

```
说明模型没有使用知识库
```

---

### 方法2：加入错误 context

例如：

```
context: 退货政策是30天
```

如果模型仍然回答：

```
7天
```

说明模型可能依赖 **预训练知识**。

---

## 4 行为指标分析

在真实系统中，还可以观察：

- **Grounded Answer Rate**
- **Hallucination Rate**

这些指标可以反映：

```
模型是否依赖知识库
```

---

## 面试总结

> 判断模型是否真正使用知识库通常可以通过三种方法：首先检查回答是否引用正确文档，其次通过 groundedness 或 faithfulness 评测回答是否被上下文支持，最后可以通过对抗测试，例如移除或修改上下文，观察模型回答是否发生变化。这些方法结合起来可以判断模型是否真正依赖知识库。
> 

# 如果知识库更新了，你怎么验证 RAG 系统真的学到了新知识？

这是 **RAG 系统评测里非常典型的问题**。核心思路一般是：

> **通过设计针对新知识的评测数据，并结合对比实验，验证模型是否真的依赖更新后的知识库。**
> 

我通常会从 **三步验证**来回答：构造评测集、对比实验、行为验证。

---

## 1 构造“新知识”评测集（Knowledge Update Test Set）

当知识库更新后，首先需要构造一批 **专门针对新知识的问题**。

例如：

知识库更新：

```
退款政策从 7 天 → 3 天
```

构造评测问题：

```
签收后多久可以申请退款？
```

这些问题的特点是：

```
只有新知识库里才有正确答案
```

---

## 2 做对比实验（Before vs After）

然后做 **更新前 vs 更新后** 的对比评测。

例如：

| 系统版本 | 回答 |
| --- | --- |
| 更新前 | 7天 |
| 更新后 | 3天 |

如果更新后系统能正确回答：

```
说明新知识被使用
```

---

## 3 验证模型是否依赖知识库

还需要确认模型不是：

```
猜对答案
```

可以通过 **对抗测试**：

### 方法1：移除知识库

```
question + 无context
```

如果模型仍然回答：

```
3天
```

说明：

```
模型没有依赖知识库
```

---

### 方法2：修改 context

例如：

```
context: 退款政策 10 天
```

观察模型回答是否改变。

---

## 4 观察系统指标变化

在系统层面，还可以观察：

- **Grounded Answer Rate**
- **Answer Correctness**
- **Hallucination Rate**

更新后，如果这些指标在相关问题上提升，也说明：

```
系统成功使用新知识
```

---

## 面试总结

> 如果知识库更新，我通常会构造一批针对新知识的评测问题，并进行更新前后的对比评测。同时通过对抗测试，例如移除或修改 context，验证模型是否真的依赖知识库而不是猜测答案。通过这些方法可以确认 RAG 系统是否真正利用了新的知识。
> 

---

# **如果用户问题在知识库里没有答案，RAG 应该怎么做？**

这是 **RAG 系统设计与评测中非常关键的问题**。成熟的回答通常不是简单说“拒答”，而是说明 **系统应该先判断是否有知识支持，再选择合适策略**。核心思路是：

> **当知识库没有答案时，RAG 系统应该避免幻觉，并通过合理的 fallback 机制处理用户请求。**
> 

一般会有 **三种策略**。

---

## 1 拒答或说明没有相关信息（Safe Fallback）

如果检索结果中：

```
没有可靠知识
```

系统应该：

```
明确说明无法找到答案
```

例如：

> “目前知识库中没有相关信息，建议联系人工客服。”
> 

这样可以避免：

```
hallucination
```

---

## 2 引导用户补充信息

有时问题不是没有知识，而是：

```
问题过于模糊
```

例如：

用户问：

```
怎么退款？
```

系统可以：

```
询问订单信息
```

这样可以提高检索成功率。

---

## 3 转人工或调用工具

在实际业务场景中，例如客服系统，如果知识库没有答案：

系统可以：

```
转人工客服
```

或调用：

- 工单系统
- 客服系统

---

## 4 评测体系中的关注指标

在评测中通常会关注：

| 指标 | 含义 |
| --- | --- |
| Refusal Accuracy | 是否正确拒答 |
| Hallucination Rate | 幻觉率 |
| Fallback Success Rate | fallback 成功率 |

这些指标可以评估系统：

```
在没有知识时是否安全
```

---

## 面试总结

> 如果用户问题在知识库中没有答案，RAG 系统应该避免编造信息，而是通过合理的 fallback 机制处理，例如明确说明没有相关信息、引导用户补充问题或转人工客服。在评测中通常也会关注拒答准确率和幻觉率等指标，以确保系统在知识缺失情况下仍然安全可靠。
> 

---

# **你觉得 RAG 和 Agent 的评测有什么区别？**

**Agent 的任务成功率，本质上是衡量 Agent 是否真正完成了任务目标。**

和普通模型不同，Agent 是一个 **多步骤系统**，它会进行规划、调用工具并和环境交互，所以我们不能只根据模型生成的一句话来判断成功，而是要看 **任务目标是否在环境中被真正达成。**

因此在评测中，我通常会把 **任务成功**定义为：

> Agent 经过一系列行动后，环境状态达到了预先定义的目标条件，并且这个结果是可以客观验证的。
> 

举几个典型例子：

- **Coding Agent**：代码是否通过全部测试用例
- **Tool Agent**：工具调用是否成功并返回正确结果
- **Web Agent**：是否完成页面操作，例如成功创建订单
- **Assistant Agent**：用户的问题是否真正被解决

关键点是：**成功必须是可验证的，而不是 Agent 自己声称完成。**

另外，在真实评测中我通常不会只看单一的 success rate，因为 Agent 可能通过很多无效步骤才完成任务，甚至可能出现错误工具调用或幻觉。所以 success rate 通常需要结合几个指标一起看，比如：

- **任务成功率**（是否完成任务）
- **步骤效率**（用了多少步）
- **工具调用正确率**
- **错误恢复能力**

这样才能更全面地评估 Agent 的真实能力。

**总结来说：**

Agent 的任务成功率其实是在衡量 **Agent 在真实环境中完成目标任务的可靠性**，核心是看 **环境状态是否达到任务目标，而不是模型生成了什么文本。**

真正的泛化能力，往往体现在 hardest slice

> **hardest slice = 数据中对模型最难、最容易出错的那一小部分子集。**
> 

也可以理解为：

> 模型的“痛点区域”
> 
> 
> 或
> 
> 泛化能力的“压力测试区”
> 

RAG的hardest slice

- 检索命中但排序靠后
- 多文档融合
- 证据冲突
- 长文档

LLM的hardest slice：

- 长上下文
- 多跳推理
- 否定句
- 数值推理
- OOD 问题（OOD = Out-Of-Distribution（分布外）问题）
- 复杂指令

为什么验证数据增强时特别要看 hardest slice？

因为增强真正的价值通常是：

> **提升模型在困难区域的鲁棒性**
> 

如果你只看到：

- easy 提升
- overall 提升

但：

- hardest 不动甚至下降

那增强很可能只是：

- 过拟合 easy 模式
- 或分布投机

# 在做评测时如何切分数据：

至少做三层切片：

1. easy
2. medium
3. hard（或 hardest quantile）

最简单可行做法：

- 按 baseline loss 排序
- 取最差 20% 当 hardest slice

这是工业界很常见的自动化方法。

## 1 评测的核心价值：提供客观反馈

研发和产品往往关注：

- 功能实现
- 产品需求

因此需要一个独立的评测体系去回答：

```
模型到底有没有变好？
```

如果没有系统化评测，很容易出现：

- 主观判断

## 2 评测可以防止模型退化

在大模型迭代过程中，经常会出现：

```
新模型某些能力提升
但某些能力退化
```

例如：

- 推理能力提升
- 但安全性下降

评测体系可以通过：

- regression test
- benchmark
- slice analysis

及时发现这些问题。

---

## 3 评测能避免“错优化”

研发和产品在优化系统时，可能会：

- 只关注某个指标
- 或某个用户案例

这容易导致：

```
局部优化
```

评测体系可以提供：

- 多维指标
- 系统级评测

避免系统被“刷指标”。

---

## 4 评测是研发与产品之间的桥梁

评测团队通常会：

- 将用户问题转化为评测数据
- 将业务目标转化为评测指标

例如客服场景：

- 用户目标：解决问题
- 评测指标：Task Success Rate

这样可以让研发更清楚：

```
优化方向是什么
```

---

## 面试总结（推荐回答）

> 我认为评测是非常有价值的，因为 AI 系统是概率系统，很难仅通过研发或产品的主观判断来评估模型效果。评测体系可以提供客观反馈，帮助团队判断模型是否真正提升，同时通过回归测试和多维指标避免模型退化或错误优化。此外评测还能把用户需求转化为可量化指标，在研发和产品之间起到桥梁作用。

# **如果没有评测团队，公司会发生什么？**

这个问题其实是在考察你是否理解 **评测团队在 AI 研发体系中的角色**。一个成熟的回答通常不是强调“评测团队多重要”，而是说明 **没有评测体系时，研发流程会出现哪些具体问题**。

我通常会从 **四个方面**回答。

---

## 1 模型迭代会变成“拍脑袋”

如果没有评测团队，模型优化往往依赖：

- 个别案例
- 主观感觉

例如：

```
这个回答看起来更好
```

但实际上可能：

- 某些能力提升
- 另一些能力退化

没有系统评测，很难判断：

```
模型是否真的变好了
```

---

## 2 很难发现模型退化

大模型迭代过程中经常会出现：

```
A能力 ↑
B能力 ↓
```

例如：

- 推理能力提升
- 安全策略下降

如果没有 **回归评测（regression evaluation）**，这些问题可能只有在 **上线之后** 才被发现。

---

## 3 容易被指标误导（Goodhart’s Law）

如果没有评测团队设计指标，团队可能只优化：

- 点击率
- 用户满意度

模型就可能通过：

- 生成更长回答
- 迎合用户

来“刷指标”，而不是真正提升用户价值。

---

## 4 产品问题很难定位

AI 应用通常是一个系统：

```
用户输入
↓
RAG检索
↓
大模型
↓
工具调用
↓
最终回答
```

如果用户体验变差，没有评测体系就很难判断：

- 是检索问题
- 模型问题
- prompt问题
- 工具调用问题

评测体系可以通过 **系统级评测**帮助快速定位问题。

---

## 面试总结（推荐回答）

> 如果没有评测团队，模型迭代很容易变成主观判断，团队难以客观判断模型是否真的提升。同时缺乏回归评测会导致模型退化问题只能在上线后被发现。此外如果没有系统化评测指标，团队也容易被单一指标误导。对于复杂的 AI 系统来说，评测体系还承担着问题定位和质量保障的角色，因此评测团队在 AI 研发流程中是非常重要的一环。

# **评测团队会不会成为创新的阻碍？**

这是一个**很常见但有点“陷阱式”的面试问题**。好的回答既不能否认风险，也要说明 **评测团队如何设计机制支持创新而不是拖慢创新**。核心观点一般是：

> **评测团队如果设计不好，确实可能成为创新阻碍；但设计得好，反而是创新的加速器。**
> 

我通常会从三个层面回答。

---

## 1 为什么有人觉得评测会阻碍创新

在一些组织里，评测团队可能会：

- 设置过多上线门槛
- 所有改动都必须跑大量评测
- 指标过于僵化

这可能导致：

```
创新成本 ↑
实验速度 ↓
```

例如：

- 每次改 prompt 都要跑完整 benchmark
- 发布流程过于复杂

这确实可能拖慢迭代。

---

## 2 评测团队真正的角色：降低试错成本

好的评测体系其实是为了：

```
让团队更快试错
```

例如：

- 自动化评测
- 快速回归测试
- 实验对比报告

这样研发可以：

- 更快验证想法
- 更快发现问题

而不是依赖人工测试。

---

## 3 设计“支持创新”的评测体系

为了避免阻碍创新，评测体系通常会分层设计。

### （1）轻量评测（快速实验）

用于：

- prompt优化
- 小模型实验

特点：

- 小规模数据集
- 快速运行

---

### （2）完整评测（上线前）

用于：

- 模型发布
- 大版本更新

特点：

- 全量评测集
- 安全与回归检查

---

这样可以做到：

```
实验快
上线稳
```

---

## 面试总结

> 评测团队如果流程设计不合理，确实可能拖慢创新。但一个好的评测体系实际上是创新的加速器，因为它可以通过自动化评测和快速回归测试降低试错成本。通常我们会设计分层评测体系，例如快速实验评测和上线前完整评测，这样既能保证质量，又不会影响团队的创新速度。
>

# **如果研发团队认为评测指标不合理，你会怎么处理？**

这是一个**非常典型的协作问题**，本质是在考察：

**评测指标是如何与研发和业务目标对齐的**。一个成熟的处理方式通常不是坚持“指标一定对”，而是通过 **数据验证、共同定义目标和实验验证** 来解决分歧。

我通常会从 **三个步骤**处理。

---

## 1 先理解研发的具体问题

首先不会直接否定研发的意见，而是需要搞清楚：

```
为什么研发觉得指标不合理
```

常见原因包括：

- 指标不能反映真实用户体验
- 指标容易被“刷分”
- 指标和业务目标不一致

例如：

研发可能说：

> 模型回答更详细，但评测分数却下降。
> 

这说明：

```
评测指标可能没有覆盖“完整性”
```

---

## 2 用数据验证指标是否有效

接下来通常会通过 **数据分析或实验验证**：

例如：

- 对比 **评测指标 vs 用户满意度**
- 分析 **评测指标 vs 任务成功率**

如果发现：

```
评测指标 与 用户体验 不相关
```

说明指标确实需要调整。

---

## 3 与研发共同调整评测体系

如果指标存在问题，通常会：

- 修改指标定义
- 增加新的指标
- 调整权重

例如：

客服场景中可能增加：

- Task Success Rate
- Actionability（可执行性）

确保指标更贴近业务目标。

---

## 总结

> 如果研发团队认为评测指标不合理，我通常会先理解他们的具体问题，而不是直接否定。然后通过数据分析或实验验证指标是否真的与用户体验或业务目标不一致。如果指标确实存在问题，我会和研发团队一起调整评测体系，例如增加新的指标或优化指标定义，确保评测能够真实反映系统效果。
>

# **如果研发和评测团队对模型好坏的判断完全相反，你怎么办？**

这是一个典型的 **跨团队决策问题**。在真实团队里，研发和评测对模型好坏判断相反并不少见，因为双方关注点不同：

- **研发**往往关注某些能力是否提升（例如推理、生成质量）
- **评测**更关注整体系统指标、稳定性和风险

我通常会通过 **三步处理：统一目标 → 数据验证 → 实验决策**。

---

## 1 先回到共同目标（业务指标）

首先要避免陷入：

```
研发观点 vs 评测观点
```

而是回到一个更客观的问题：

```
这个模型是否更符合业务目标？
```

例如客服场景，核心目标可能是：

- Task Success Rate（任务成功率）
- 转人工率
- 用户满意度

如果模型在这些指标上没有提升，那么即使某些能力提升，也不一定适合上线。

---

## 2 分析差异来源

如果双方判断完全相反，通常说明 **评测维度不一致**。

例如：

| 团队 | 关注指标 |
| --- | --- |
| 研发 | 生成质量、推理能力 |
| 评测 | 任务成功率、系统稳定性 |

这时需要：

- 对比不同指标
- 做 slice analysis

看看模型在哪些场景：

```
变好
或
变差
```

---

## 3 用实验做最终判断

如果仍然存在争议，通常会通过：

**A/B test**

让真实用户数据来验证：

```
哪个模型表现更好
```

例如：

- 用户满意度
- 任务完成率
- 追问率

用数据来决定，而不是依赖主观判断。

---

## 总结

> 如果研发团队和评测团队对模型好坏判断完全相反，我通常会先回到共同的业务目标，而不是停留在各自的观点上。然后通过分析不同评测维度找到分歧原因，例如模型在哪些场景提升或退化。如果仍然存在争议，通常会通过 A/B test 或用户数据来做最终决策，用真实数据验证哪个模型更符合业务目标。
> 

---

# **如果线上实验也无法区分两个模型好坏，你会怎么办？**

这是一个**实验设计与决策问题**。如果 **线上 A/B 实验也无法区分两个模型**，通常说明两种可能：

- 两个模型确实 **效果非常接近**
- 当前实验 **信号不足（样本量、指标、用户分布等）**

作为评测负责人，我一般会按 **三步处理：先验证实验有效性 → 再增加区分能力 → 最后做理性决策**。

---

## 1 先检查实验是否有统计显著性

首先要确认实验是否真的“无法区分”，还是因为 **样本量不足**。

需要检查：

- 样本量是否足够
- 实验时间是否过短
- 指标波动是否过大

例如：

```
p-value > 0.05
```

可能只是 **统计功效（power）不足**。

解决方法：

- 延长实验时间
- 增加流量
- 提高样本量

---

## 2 提升实验的区分能力

如果两个模型在整体指标上接近，可以尝试 **更细粒度分析**。

例如：

**Slice analysis**

按不同场景分析：

| Slice | 例子 |
| --- | --- |
| 退款问题 | 退货规则 |
| 物流问题 | 订单查询 |
| 复杂问题 | 多轮对话 |

有时会发现：

```
某些场景差异明显
```

而 overall 指标被平均掉了。

---

## 3 引入更多评测信号

如果单一指标无法区分，可以加入更多信号，例如：

- follow-up rate（追问率）
- conversation turns（对话轮数）
- abandon rate（放弃率）

这些行为指标有时比单一满意度更敏感。

---

## 4 最后做产品决策

如果经过分析仍然发现：

```
两个模型表现几乎一致
```

那其实也是一个结果。

这时可以根据其他因素决策，例如：

- 成本（推理成本）
- 延迟（latency）
- 系统稳定性

选择更优方案。

---

## 面试总结

> 如果线上实验无法区分两个模型，我通常会先检查实验是否具有足够的统计功效，例如样本量和实验时间是否足够；其次通过 slice analysis 或更多行为指标提高实验的区分能力。如果经过分析仍然没有明显差异，我会认为两个模型在用户体验上基本等价，这时可以根据成本、延迟或系统稳定性等因素做最终决策。
>

# **你觉得 AI 评测和传统软件测试最大的区别是什么？**

这是一个**非常经典的 AI 评测面试题**。核心差异其实在于：

> **传统软件是确定性系统，而 AI 系统是概率系统。**
> 

因此评测方法、指标设计和测试方式都会完全不同。我一般会从 **三个关键差异**来回答。

---

## 1 确定性 vs 概率性

传统软件通常是 **确定性系统**：

```
输入 A → 一定输出 B
```

例如：

- API 返回结果
- 业务规则计算

测试方式通常是：

- unit test
- integration test

只要输出符合预期就是 **pass**。

---

AI 系统则是 **概率生成系统**：

```
同一个输入 → 可能产生多个合理输出
```

例如：

用户问：

```
怎么保持健康？
```

可能有多种合理回答。

因此 AI 评测通常需要：

- 评分
- 排序
- 概率统计

而不是简单的 **pass / fail**。

---

## 2 测试目标不同

传统软件测试关注：

- 功能是否正确
- 是否存在 bug

AI 评测则需要关注更多维度，例如：

- Correctness（正确性）
- Helpfulness（实用性）
- Safety（安全性）
- Robustness（鲁棒性）

很多指标还具有 **主观性**。

---

## 3 测试对象不同

传统软件通常测试的是：

```
代码逻辑
```

而 AI 应用往往是一个 **复杂系统**：

```
用户输入
↓
RAG 检索
↓
大模型
↓
工具调用
↓
最终回答
```

因此 AI 评测不仅要评模型，还要评：

- 数据
- prompt
- 检索系统
- 工具调用

属于 **系统级评测**。

---

## 总结

> AI 评测和传统软件测试最大的区别在于系统性质不同。传统软件是确定性系统，可以通过明确的输入输出进行 pass/fail 测试；而 AI 系统是概率系统，同一个输入可能产生多个合理输出，因此评测通常需要使用评分和统计方法。此外 AI 应用通常是一个复杂系统，不仅涉及模型，还涉及数据、检索和工具调用，因此评测更偏向系统级评估，而不是单纯的功能测试。
> 

# 为什么传统的机器学习模型没有专门的评测岗位，现在大模型及其应用需要评测岗位

这个问题在面试里其实是在考察你是否理解 **大模型时代评测岗位出现的“结构性原因”**。核心逻辑一般是：

> **传统机器学习问题相对封闭、指标清晰，而大模型任务开放、指标复杂，因此需要专门的评测体系和团队。**
> 

我通常会从 **四个方面解释这个变化。**

---

## 1 任务类型不同：封闭任务 vs 开放任务

传统机器学习模型通常解决的是 **封闭任务（closed tasks）**，例如：

- 分类（classification）
- 回归（regression）
- 排序（ranking）

这些任务通常有 **明确的正确答案**：

```
spam / not spam
```

评测指标也很明确：

- Accuracy
- Precision / Recall
- AUC

因此评测通常可以：

```
自动计算指标
```

不需要单独团队。

---

而大模型处理的是 **开放任务（open-ended tasks）**：

- 对话
- 写作
- 总结
- 推理

例如：

```
解释什么是量子计算
```

没有唯一正确答案，因此评测变得：

- 更主观
- 更复杂

需要：

- LLM-as-judge
- 人工评测
- 多维指标

---

## 2 系统复杂度不同

传统 ML 通常评测的是：

```
单个模型
```

而大模型应用往往是一个系统：

```
用户输入
↓
RAG 检索
↓
大模型
↓
工具调用
↓
最终回答
```

问题可能来自：

- 检索系统
- prompt
- 模型
- 工具调用

因此需要 **系统级评测**。

---

## 3 风险与安全问题更复杂

传统 ML 风险相对有限，例如：

- 分类错误

而大模型可能产生：

- 幻觉
- 有害内容
- 隐私泄露

因此需要专门的评测去检测：

```
安全性
```

---

## 4 产品迭代方式不同

传统 ML 模型更新频率通常较低，例如：

```
几周或几个月
```

而大模型应用往往：

- prompt 调整
- RAG 更新
- 模型升级

更新非常频繁。

如果没有自动评测体系，很容易：

```
模型退化
```

---

## 面试总结

> 传统机器学习模型通常解决的是封闭任务，例如分类或回归，这些任务有明确的正确答案和成熟的自动评测指标，因此评测通常由研发团队完成。而大模型处理的是开放任务，例如对话和生成，没有唯一正确答案，同时应用系统也更复杂，还涉及安全和合规问题。因此需要专门的评测体系和评测团队来设计指标、构建评测数据并持续监控模型表现。
> 

# 你觉得未来 AI 评测岗位会不会消失？

这是一个**趋势判断问题**。一个比较成熟的回答通常不是简单说“不会”，而是说明 **评测岗位会发生什么变化**。核心观点一般是：

> **AI 评测岗位不会消失，但角色会从“人工测试”转变为“评测体系设计者”。**
> 

我通常从 **三个方面解释。**

---

## 1 一部分评测会被自动化

随着技术发展，很多评测工作已经开始自动化，例如：

- **LLM-as-judge**
- 自动 benchmark
- 自动回归测试

这些工具可以自动完成：

```
基础评测
```

例如：

- 正确性评分
- 回答质量评分

因此未来评测岗位不会主要做：

```
人工打分
```

---

## 2 评测体系设计反而更重要

虽然评测执行可以自动化，但**评测体系设计**仍然需要人。

例如需要回答：

- 评测什么指标？
- 指标是否反映用户价值？
- 如何避免模型刷指标？

这些问题属于：

```
evaluation design
```

而不是简单测试。

---

## 3 AI 系统越来越复杂

未来 AI 系统通常包含：

```
模型
RAG
工具调用
Agent
```

评测对象已经从：

```
单模型
```

变成：

```
复杂 AI 系统
```

这反而需要更系统化的评测能力，例如：

- system evaluation
- safety evaluation
- alignment evaluation

---

## 面试总结

> 我认为 AI 评测岗位不会消失，但角色会发生变化。随着 LLM-as-judge 和自动化评测工具的发展，一部分评测执行工作会被自动化。但评测体系设计、指标定义以及系统级评估会变得更加重要。特别是在 AI 应用越来越复杂的情况下，评测岗位需要从简单测试转变为评测体系和质量治理的设计者。
> 

# 如果 AI 自己可以评测 AI，人类评测还有必要吗？

这是一个**关于 LLM-as-judge 局限性的问题**。一个成熟的回答通常是：

> **AI 可以大幅提高评测效率，但人类评测仍然不可替代，因为 AI 评测本身也需要人类校准和监督。**
> 

我一般从 **三个方面解释。**

---

## 1 AI 评测仍然可能存在偏差

LLM-as-judge 在很多场景中很有效，但仍然可能存在 **系统性偏差（bias）**，例如：

- **Position bias**：更容易偏好第一个回答
- **Length bias**：更容易给更长回答更高分
- **Style bias**：偏好某种表达风格

如果完全依赖 AI 评测，可能会导致：

```
模型优化评测器
```

而不是真正提升能力。

---

## 2 AI 很难理解真实用户价值

AI 可以评测：

- 语言质量
- 逻辑结构

但很难完全理解：

```
真实用户需求
```

例如客服场景中：

用户更关注：

- 是否解决问题
- 是否符合业务规则

这些往往需要 **业务专家**参与评估。

---

## 3 AI 评测需要人类校准

在实际评测体系中，常见做法是：

```
AI评测 + 人类抽检
```

流程通常是：

1️⃣ AI 自动评测大规模数据

2️⃣ 人类评审关键样本

3️⃣ 用人工结果校准 AI judge

这样可以同时获得：

- **效率**
- **可靠性**

---

## 面试总结

> 即使 AI 可以评测 AI，人类评测仍然是必要的。LLM-as-judge 虽然可以大幅提高评测效率，但仍然可能存在偏差，例如位置偏差或长度偏差。同时 AI 很难完全理解真实用户需求，因此在人类评测中通常需要业务专家参与。实际评测体系通常采用 AI 自动评测结合人工抽检和校准的方式，以兼顾效率和可靠性。
>
