# AI评测的难点

**第一，非确定性输出（Non-deterministic outputs）。**

大模型是概率生成模型，即使输入相同，在不同 sampling 参数或随机种子下也可能生成不同结果。因此评测不能只看单次输出，通常需要 **多次采样、统计指标（pass@k、平均得分等）** 来获得稳定的评测结论。

**第二，强数据依赖（Data dependency）。**

模型表现高度依赖数据，包括两个层面：

- **外部数据**：例如 RAG 系统中检索库或知识库的质量直接影响最终回答。
- **模型数据**：模型版本、训练数据分布变化都会导致评测结果波动。
    
    因此评测必须同时控制 **数据版本、模型版本和知识库版本**，否则结果难以复现。
    

**第三，可解释性弱（Lack of interpretability）。**

大模型本质上是黑盒模型，内部推理过程难以直接观察。如果没有 **CoT（Chain-of-Thought）或中间推理轨迹**，很难判断模型为什么产生某个答案，也不容易定位错误来源，这会增加调试和评测的难度。

**第四，伦理与合规风险（Ethics & compliance）。**

大模型可能产生 **偏见、隐私泄露或不当内容**，因此评测不仅要看能力指标，还需要评估

- fairness / bias
- safety
- privacy
- 合规性
    
    这类评测往往需要专门的安全测试集和人工审核。
    

**总结来说：**
AI评测的核心难点在于，大模型是**非确定性的概率生成系统**，其表现又**高度依赖数据和模型版本**，同时由于模型的**黑盒特性导致可解释性不足**，再加上需要满足**安全、伦理与合规要求**，因此评测不仅是指标计算问题，更是一个涉及**统计稳定性、数据治理、可解释性和安全治理的系统工程**。

# LLM评测范围

从工业实践来看，LLM 的评测范围通常覆盖 **四个层面：模型能力、系统能力、安全合规以及产品体验**。

**第一，模型能力评测（Model capability）。**

主要评估模型本身的智能水平，例如：

- 知识与事实能力（factuality）
- 推理能力（reasoning / multi-hop reasoning）
- 指令跟随（instruction following）
- 代码与数学能力

常见 benchmark 包括 **MMLU、GSM8K、HumanEval** 等。

**第二，系统能力评测（System evaluation）。**

在真实应用中，大模型往往是系统的一部分，比如 **RAG、Agent、Tool use**。

因此需要评测：

- 检索质量（recall / precision）
- 工具调用成功率
- 端到端任务成功率
- latency 和稳定性

**第三，安全与对齐评测（Safety & alignment）。**

这是顶级 AI 公司非常重视的部分，需要评测模型是否：

- 产生有害内容（toxicity）
- 存在偏见（bias）
- 泄露隐私或敏感信息
- 能否抵抗 prompt injection / jailbreak

**第四，产品体验评测（Product quality）。**

从用户角度评估：

- relevance（是否回答问题）
- helpfulness（是否有帮助）
- hallucination rate
- 用户满意度

通常结合 **LLM-as-judge + 人工评测**。

**一句话总结**

> 在工业实践中，LLM 评测通常覆盖四个维度：**模型能力、系统能力、安全对齐以及产品体验**，通过 **benchmark、自动指标、LLM-as-judge 和人工评测** 的组合来进行全面评估。
> 

# 为什么传统 NLP benchmark 已经不够评测 LLM？

传统 NLP benchmark 在早期模型时代是有效的，但对于 LLM 来说已经不够，主要有四个原因：

**第一，任务形式过于封闭（Closed-form tasks）。**

传统 benchmark 大多是 **分类或单一标准答案任务**，例如 sentiment classification 或 multiple-choice QA。但 LLM 的主要能力是 **开放式生成和复杂推理**，这种能力很难通过固定答案的 benchmark 完整衡量。

**第二，benchmark 容易被训练数据污染（Data contamination）。**

当前大模型的训练语料规模非常大，很多 benchmark（如 SQuAD、GLUE 等）可能已经出现在训练数据中，因此高分不一定代表真实能力，而可能只是 **记忆或数据泄露**。

**第三，无法评估真实应用场景（Lack of real-world evaluation）。**

传统 benchmark通常是 **单轮、单任务评测**，但现实应用中 LLM 需要处理：

- 多轮对话
- 长上下文
- RAG 检索
- tool / agent 调用

这些系统级能力无法通过传统 NLP benchmark 衡量。

**第四，缺乏安全与对齐评测（Safety & alignment）。**

现代 LLM 需要满足 **安全性、偏见控制、合规性等要求**，例如 jailbreak、toxicity、prompt injection 等问题，这些维度在传统 benchmark 中基本没有覆盖。

**总结来说：**

> 传统 NLP benchmark 主要评测“单任务准确率”，而 LLM 的能力更像是一个 **开放式、系统级智能体**，因此评测需要从 **能力评测扩展到系统评测、安全评测以及真实用户场景评测**。
> 

---

**顶级 AI 面试一句话版本：**

> 传统 NLP benchmark 的局限在于 **封闭任务形式、训练数据污染、缺乏真实系统场景以及缺少安全评测**，而 LLM 的能力是开放式生成和系统级智能，因此需要更接近真实应用的综合评测体系。
> 

# RAG切分有什么注意点

在 RAG 系统中，文档切分（chunking）对检索质量和最终回答质量影响非常大。通常需要重点考虑四个方面：

**第一，语义完整性（semantic coherence）。**

切分时要尽量保证每个 chunk 语义完整，例如按 **段落、标题、章节结构** 切分，而不是简单按固定 token 数量截断，否则可能导致关键信息被拆散，影响检索和生成。

**第二，chunk size 与模型上下文窗口匹配。**

chunk 太小会导致信息碎片化，检索结果缺乏上下文；

chunk 太大又会降低检索精度，并增加上下文占用。

在实践中通常会选择 **200–500 tokens** 左右，并根据模型 context window 和文档类型进行调整。

**第三，chunk overlap（重叠）。**

为了避免关键信息被边界截断，一般会设置 **一定的 overlap（例如 10–20%）**，这样可以提高召回率，但也需要控制重叠比例，否则会增加向量库规模和检索冗余。

**第四，结构化信息保留（metadata & structure）。**

在切分时要保留重要结构信息，例如：

- 标题层级
- 文档来源
- 时间 / 作者
- section ID

这些 metadata 可以用于 **filter 检索、rerank 或引用来源**。

---

**总结一句话（面试加分版）：**

> RAG 的文档切分核心是在 **语义完整性、chunk size、overlap 和结构化 metadata** 之间取得平衡，从而同时优化 **检索召回率、检索精度以及上下文利用效率**。
>

# 数据最好在哪个阶段处理？

越早越好，从收集阶段最好！！

数据通常**越早处理越好**，最佳阶段是在 **数据进入训练或推理流程之前的数据预处理阶段（data preprocessing / data pipeline stage）**。

原因主要有三个：

**第一，提高数据质量，避免垃圾数据进入模型。**

如果在训练或评测阶段才处理数据，错误或噪声数据已经影响模型学习，因此需要在 **数据 ingestion 或 preprocessing 阶段**进行清洗，例如：

- 去重（deduplication）
- 噪声过滤
- 格式统一
- 敏感信息清理

**第二，提高数据一致性和可复现性。**

在数据 pipeline 的早期进行标准化处理，可以保证 **训练数据、评测数据和推理数据使用同一套处理规则**，从而保证实验可复现。

**第三，降低系统复杂度。**

如果把数据处理分散在训练或模型逻辑中，会增加系统复杂度；将其集中在 **ETL / data pipeline** 中可以让模型训练和推理流程更加清晰。

---

**总结：**

> 在工业实践中，数据最好在 **数据进入训练或推理流程之前的数据预处理阶段统一处理**，这样可以同时保证 **数据质量、系统一致性以及实验可复现性**。

# 金标数据是什么，哪里用到?

**金标数据（Gold Standard Data / Gold Labels）指的是经过专家或高质量人工严格标注、被认为最可靠的标准答案数据**，通常作为评测或质量控制的基准。

## 一、金标数据是什么

简单来说：

> 金标数据就是 **被认为最准确、最可信的参考答案数据（ground truth）**。
> 

它通常具有几个特点：

- 由 **专家或高质量标注员**标注
- 标注规则严格
- 经过 **多轮审核（review / adjudication）**
- 质量高于普通标注数据

因此可以作为：

```
benchmark
evaluation reference
quality control baseline
```

## 二、金标数据主要用在哪里

### 1 模型评测（Evaluation）

最常见用途。

例如：

```
question
reference answer (gold label)
model answer
```

通过比较：

- EM
- F1
- accuracy

评估模型性能。

### 2 标注质量控制（Annotation QA）

在大规模标注任务中，通常会混入 **gold tasks**。

例如：

标注员任务：

```
task1
task2
task3
task4 (gold)
```

如果标注员在 gold task 上错误率高：

```
说明标注质量有问题
```

### 3 模型训练（Supervised Learning）

在监督学习中：

```
input → gold label
```

模型通过 gold label 学习正确输出。

例如：

- 分类任务
- QA任务
- instruction tuning

### 4 RLHF / 偏好训练

在大模型训练中，gold 数据可以用于：

- **reward model 训练**
- **偏好数据校准**

例如：

```
response A
response B
gold preference
```

## 三、为什么金标数据很重要

因为模型评测本质上依赖：

```
reference answer
```

如果参考答案质量低：

```
evaluation 不可信
```

因此在工业实践中：

- **评测集通常必须是 gold dataset**

## 一句话总结

> 金标数据是经过专家或高质量人工标注、被认为最可靠的标准答案数据，通常用于 **模型评测、标注质量控制、监督学习训练以及 RLHF 偏好数据构建**。
> 

# 你觉得为什么要落地在客服场景

大模型优先落地在**智能客服场景**，主要有三个原因：**需求匹配度高、商业价值明确、技术可控性强**。

**第一，需求与大模型能力高度匹配。**

客服场景本质是**自然语言理解和生成任务**，例如：

- 用户问题理解
- FAQ问答
- 多轮对话
- 问题总结与解释

这些正是大模型最擅长的能力，因此落地门槛相对较低。

**第二，商业价值清晰且 ROI 明确。**

客服是企业成本较高的环节，大模型可以：

- 自动回答高频问题
- 提升客服效率
- 减少人工成本

因此企业更容易看到 **明确的降本增效价值**。

**第三，场景边界相对可控。**

相比开放领域应用，客服通常：

- 有明确知识库（FAQ、产品文档）
- 可以结合 **RAG 检索增强**
- 问题类型相对集中

这样可以降低幻觉风险，提高系统可靠性。

**一句话总结：**

> 智能客服之所以成为大模型优先落地场景，是因为它与大模型的语言能力高度匹配，同时具备明确的商业价值和相对可控的知识边界，便于通过 RAG 等技术实现可靠落地。
> 

# 你用过哪些评测指标？他们的使用场景是什么？

在实际项目中，我使用过的评测指标主要分为 **四类：分类任务指标、生成任务指标、检索系统指标以及大模型质量评测指标**，不同任务会选择不同指标。

**1 分类任务指标（Classification Metrics）**

常见指标：

- **Accuracy（准确率）**
- **Precision（精确率）**
- **Recall（召回率）**
- **F1 Score（F1值）**

使用场景：

- 意图识别（Intent Classification）
- 情感分析（Sentiment Analysis）
- 内容审核（Toxicity Detection）

说明：

- Precision 关注误报率
- Recall 关注漏报率
- F1 综合两者

**2 文本生成指标（Text Generation Metrics）**

常见指标：

- **BLEU（双语评估替换指标）**
- **ROUGE（召回导向摘要评估指标）**
- **METEOR（机器翻译评估指标）**

使用场景：

- 机器翻译
- 文本摘要
- 自动生成文本

这些指标通过 **n-gram 重叠度** 来衡量生成文本与参考答案之间的相似程度。

**3 检索系统指标（Retrieval Metrics）**

在 **RAG 或搜索系统**中经常使用。

常见指标：

- **Precision@K（K位精确率）**
- **Recall@K（K位召回率）**
- **MRR（Mean Reciprocal Rank，平均倒数排名）**
- **NDCG（Normalized Discounted Cumulative Gain，归一化折损累计增益）**

使用场景：

- 文档检索
- 搜索排序
- RAG检索模块评测

这些指标主要评估：

- 检索结果相关性
- 正确答案是否排在前面

**4 大模型质量评测指标（LLM Evaluation Metrics）**

在大模型应用中，还会评估生成质量，例如：

- **Correctness（正确性）**
- **Relevance（相关性）**
- **Helpfulness / Usefulness（实用性）**
- **Completeness（完整性）**
- **Consistency（一致性）**
- **Fluency（流畅性）**
- **Instruction Following（指令遵循能力）**
- **Hallucination Rate（幻觉率）**
- **Faithfulness / Groundedness（事实一致性）**
- **Safety（安全性）**

使用场景：

- 对话系统
- RAG系统
- Agent系统

通常通过：

- **人工评测（Human Evaluation）**
- **LLM-as-judge**
- **任务成功率（Task Success Rate）**

来综合评估。

**一句话总结：**

> 在实际项目中，我会根据任务类型选择不同评测指标，例如分类任务使用 Accuracy 和 F1，生成任务使用 BLEU 和 ROUGE，检索系统使用 Recall@K 和 MRR，而在大模型应用中还需要结合 Correctness、Relevance、Hallucination Rate、Faithfulness 和 Safety 等多维度质量评估指标。
> 

# 这些指标中，哪些可以自动评测，哪些必须人工评测？

在大模型评测中，不同指标的评测方式不同，大致可以分为 **自动评测、半自动评测和人工评测**三类。

**1 自动评测指标（Automatic Metrics）**

这些指标 **可以通过算法直接计算，不需要人工参与**，通常用于 **有标准答案（ground truth）的任务**。

常见指标：

- **Accuracy（准确率）**
- **Precision（精确率）**
- **Recall（召回率）**
- **F1 Score（F1值）**

使用场景：

- 分类任务
- 意图识别
- 情感分析
- 内容审核

此外，一些生成任务指标也可以自动计算：

- **BLEU（双语评估替换指标）**
- **ROUGE（召回导向摘要评估指标）**

适用于：

- 机器翻译
- 文本摘要

**2 半自动评测指标（LLM-assisted Metrics）**

有些指标 **可以自动计算，但通常需要模型辅助判断**，因此属于半自动评测。

常见指标：

- **Hallucination Rate（幻觉率）**
- **Faithfulness / Groundedness（事实一致性）**
- **Answer Relevance（答案相关性）**

常见方法：

- **NLI模型判断**
- **LLM-as-judge**

使用场景：

- RAG系统
- QA系统
- 对话系统

**3 人工评测指标（Human Evaluation）**

一些指标 **主观性较强，很难完全自动化评测**，通常需要人工评测。

常见指标：

- **Helpfulness / Usefulness（实用性）**
- **Completeness（完整性）**
- **Fluency（流畅性）**
- **Creativity（创造性）**
- **Safety（安全性，复杂场景）**

这些指标通常通过 **评分规则（rubric）** 进行人工打分，例如：

| 维度 | 评分 |
| --- | --- |
| 正确性 | 0-2 |
| 相关性 | 0-2 |
| 完整性 | 0-2 |

**总结：**

> 在大模型评测中，**分类任务指标如 Accuracy 和 F1 可以完全自动评测；生成任务指标如 BLEU 和 ROUGE 也可以自动计算；而像 Hallucination、Relevance、Faithfulness 等指标通常需要 LLM-as-judge 辅助评测；对于 Helpfulness、Completeness、Creativity 等主观质量指标，则通常需要人工评测。**
> 

# 你如何将用户指标和评测指标进行对齐？

在实际项目中，将用户体验与评测指标对齐通常需要 **从用户目标出发，将主观体验拆解为可量化指标**。一般可以分为三个步骤。

**1 从用户目标定义北极星指标（North Star Metric）**

首先需要明确用户在这个产品中的核心目标，例如：

智能客服场景：

用户目标：

- 快速获得正确答案
- 问题被解决

对应的北极星指标：

- **Task Success Rate（任务成功率）**
- **User Satisfaction（用户满意度）**

**2 将用户体验拆解为可评测维度**

用户体验通常可以拆解为几个质量维度，例如：

| 用户体验 | 评测维度 |
| --- | --- |
| 答案正确 | Correctness（正确性） |
| 回答相关 | Relevance（相关性） |
| 有帮助 | Helpfulness（实用性） |
| 回答完整 | Completeness（完整性） |
| 表达清晰 | Fluency / Clarity（流畅性 / 清晰度） |

这样就可以把 **主观体验转化为结构化评价维度**。

**3 将评价维度映射为具体指标**

最后将这些维度转化为可以评测的指标，例如：

| 评测维度 | 评测指标 |
| --- | --- |
| 正确性 | Accuracy / Human correctness score |
| 相关性 | Answer relevance score |
| 实用性 | Helpfulness score |
| 事实性 | Hallucination rate / Faithfulness |

评测方式：

- 自动评测
- LLM-as-judge
- 人工评测

**一个例子（智能客服）**

用户体验：

> 用户希望问题能被快速正确解决
> 

可以转化为：

评测指标：

- **Task Success Rate（任务成功率）**
- **Correctness（正确性）**
- **Answer Relevance（答案相关性）**
- **Latency（响应时间）**

**总结**

> 在实际项目中，我通常会先从用户目标出发定义北极星指标，然后将用户体验拆解为多个质量维度，例如正确性、相关性和实用性，再把这些维度映射为可量化的评测指标，例如任务成功率、幻觉率或答案相关性，从而实现用户体验和评测体系的对齐。
> 

# 如果评测指标提升了，但用户满意度没有提升，你会怎么分析？

如果评测指标提升了，但用户满意度没有提升，我通常会从 **评测指标、数据分布、产品体验和系统因素**四个方面进行分析。

**1 评测指标是否真正代表用户体验**

首先需要检查 **当前评测指标是否与用户目标一致**。

例如：

如果只优化了：

- **BLEU / ROUGE**

但用户更关心的是：

- **是否解决问题（Task Success Rate）**
- **回答是否有帮助（Helpfulness）**

那么指标提升并不一定会提升用户体验。

因此需要重新检查：

> 当前指标是否真正反映用户价值。
> 

**2 评测数据是否代表真实用户场景**

离线评测数据往往来自：

- benchmark
- 内部构造数据

但真实用户问题可能：

- 更长尾
- 更复杂
- 更口语化

如果评测数据和真实流量分布不同，就会出现：

> offline metrics 提升，但线上体验没有变化。
> 

因此需要分析：

- **线上 query 分布**
- **评测数据覆盖度**

**3 是否存在系统层面的瓶颈**

有时候用户体验不只是模型能力问题，还包括：

- **响应延迟（latency）**
- **系统稳定性**
- **检索质量（RAG）**
- **工具调用失败**

例如：

模型回答质量提升，但 **响应时间从 2 秒变成 6 秒**，用户体验反而下降。

**4 是否出现指标“过拟合”**

如果模型针对评测指标优化过度，可能会出现：

- 模型更符合 benchmark
- 但真实回答变得更机械

例如：

- overly verbose answers
- overly safe responses

这种情况说明模型 **过拟合评测指标，而不是用户需求**。

**总结**

> 如果离线评测指标提升但用户满意度没有提升，我会重点检查三个方面：第一是评测指标是否真正反映用户价值；第二是评测数据是否代表真实用户分布；第三是系统层面的因素，例如延迟或检索质量。很多时候问题并不在模型能力，而是在评测体系和真实用户需求之间存在偏差。
> 

# 如果没有用户满意度数据，你怎么验证你的评测指标是有效的？

如果没有直接的用户满意度数据，我通常会通过 **四种方法验证评测指标的有效性**。

**1 相关性分析（Correlation Analysis）**

首先可以分析 **离线指标与用户行为信号之间的相关性**。

例如：

比较不同模型版本：

| 模型版本 | 离线指标 | 用户行为 |
| --- | --- | --- |
| A | 指标较低 | 用户停留时间短 |
| B | 指标更高 | 用户继续提问减少 |

如果离线指标提升时，用户行为指标也改善，说明指标具有 **预测用户体验的能力**。

常用行为信号包括：

- 会话长度（conversation length）
- 用户追问率
- 用户放弃率

**2 在线实验（A/B Testing）**

将不同模型版本进行 **A/B 测试**。

例如：

- 模型 A：旧版本
- 模型 B：优化版本

观察：

- 用户继续使用率
- 任务完成率
- 用户留存率

如果离线指标更好的模型在线表现也更好，就说明指标是有效的。

**3 使用代理指标（Proxy Metrics）**

在没有满意度评分的情况下，可以使用 **用户行为作为代理指标**。

例如：

| 代理指标 | 含义 |
| --- | --- |
| Follow-up rate（追问率） | 回答是否解决问题 |
| Abandon rate（放弃率） | 回答是否无用 |
| Task completion rate（任务完成率） | 用户是否达成目标 |

这些指标可以间接反映用户体验。

**4 案例分析（Qualitative Analysis）**

对模型输出进行 **人工案例分析**：

- 抽样对比不同模型回答
- 检查失败案例
- 分析指标变化是否对应质量变化

如果指标提升但案例质量没有变化，就说明指标可能存在问题。

**总结：**

> 如果没有直接的用户满意度数据，我通常会通过 **相关性分析、A/B 实验、用户行为代理指标以及案例分析** 来验证评测指标的有效性，从而确保离线指标能够真实反映用户体验。
> 

# 如果不同指标之间存在 trade-off（例如正确性高但流畅性下降），你如何设计评分避免模型作弊？

指标trade off：两个指标之间天然存在优化对立关系（提升一个指标通常会降低另一个指标。）

在顶级 AI 公司里，这题的关键点是：**综合评分不能被“刷分”**。当指标存在 trade-off（正确性 vs 流畅性、helpfulness vs safety 等），我会用“**硬约束 + 非线性聚合 + 反作弊校验**”的组合来设计评分。

## 1) 先把“不能牺牲的维度”做成硬门槛（gating）

避免模型用某一项极高去掩盖另一项极低。

- **安全性（Safety）/合规（Compliance）**：硬门槛，未达标直接 fail 或强惩罚
- **正确性（Correctness）**：在某些场景（医疗/金融/客服）也可以设门槛，比如 correctness < 0.8 的样本不计入整体“优秀”分

这样可以防止出现：

“流畅但胡说” 或 “很安全但全拒答” 仍拿高分。

## 2) 聚合不用简单加权和，改用“惩罚短板”的非线性方式

加权和最大的问题：**可补偿性太强**（Compensability）。模型可能“把某一项刷到很高”来抵消另一项的下降。

更稳健的做法：

### A. 几何平均 / 调和平均（penalize low scores）

- **几何平均**：任何一项低，整体明显下降
- **调和平均**：对低分更敏感（类似 F1 的思想）

直觉：让综合分更像“木桶”，而不是“平均”。

### B. 分段惩罚（piecewise penalty）

例如流畅性低于某阈值（比如 < 3/5）就触发额外扣分，避免模型为了 correctness 输出极端难读的答案。

## 3) 用“分层评分”避免局部最优：先正确，再好读

针对 trade-off 的排序偏好，采用**字典序（lexicographic）/ 多阶段优化**：

1. 先比较 **Correctness / Safety / Instruction Following**
2. 再比较 **Relevance / Helpfulness / Completeness**
3. 最后比较 **Fluency / Style**

这能显式表达产品偏好：**宁可慢一点/不那么优雅，也不能错**；但在正确前提下再提升表达。

## 4) 做反作弊：增加“对抗样本 + 诊断指标”

模型“作弊”往往表现为：

- 过度 verbose（堆字显得更有帮助）
- 过度保守（大量拒答以提升 safety）
- 模板化回答（提升一致性/流畅性，但信息量低）

因此我会加一些**专门的反作弊诊断**，并纳入评分或门槛：

- **拒答率（Refusal Rate）**：安全提升但拒答率异常上升 → 扣分或触发 review
- **信息密度（Information Density）**：同等正确性下，过长但低信息量 → 扣分
- **格式/约束通过率（Schema / Instruction Compliance Rate）**：防止用“好看文本”规避结构化要求
- **长尾/对抗集表现（Hardest slice / adversarial set）**：防止只在常规集刷分

## 5) 用“样本级聚合”而不是“指标整体平均”

另一个常见作弊点：模型在简单样本上拿高分，掩盖困难样本掉分。

做法：

- **分桶加权（bucketed scoring）**：按难度/场景/风险分桶（简单、复杂、长上下文、高风险），每桶设最低要求
- **worst-slice 指标**：看最差 10% 样本的分数（或 P10/P25），不只看 mean

## 6) 用人类偏好或在线代理信号校准综合分

最后我会用少量高质量人工偏好（pairwise preference）或在线代理信号做校准：

- 检查综合分排序是否与人类选择一致
- 若不一致，调整门槛、非线性聚合方式或权重

## 总结：

> 面对指标 trade-off，我会用 **硬门槛（Safety/Correctness gating）+ 惩罚短板的非线性聚合（几何/调和平均或分段惩罚）+ 分层比较（先正确后表达）+ 反作弊诊断（拒答率、信息密度、hardest-slice）** 来设计综合评分，确保模型无法通过“刷某一项”掩盖关键能力退化。
> 

# **如果评测指标之间发生冲突（例如安全性提高但帮助性下降），你怎么处理？**

指标冲突：在某个模型版本或实验中，不同指标给出了相反的评价。

当评测指标之间发生冲突，例如 **安全性（Safety）提高但帮助性（Helpfulness）下降**，我通常会从三个方面处理。

**1 明确产品目标与指标优先级**

首先需要根据产品定位确定 **指标优先级（metric hierarchy）**。

例如：

在大模型产品中通常有：

**硬性约束（Hard Constraints）**

- Safety（安全性）
- 合规性（Compliance）

**优化目标（Optimization Metrics）**

- Helpfulness（实用性）
- Relevance（相关性）
- Completeness（完整性）

也就是说：

> 安全性通常是 **必须满足的底线指标**，在此基础上再优化帮助性。
> 

**2 通过权衡分析找到最优平衡点**

在实际优化中，可以通过实验找到 **Pareto optimal trade-off（帕累托最优点）**。

例如：

比较不同模型版本：

| 模型版本 | Safety | Helpfulness |
| --- | --- | --- |
| A | 高 | 低 |
| B | 中 | 高 |
| C | 高 | 中 |

通常会选择：

> **在满足安全要求的前提下帮助性最高的模型版本**。
> 

**3 通过技术手段减少指标冲突**

很多时候指标冲突可以通过技术手段缓解，例如：

- **更细粒度的安全策略**，避免过度拒答
- **更好的 prompt / policy 设计**
- **通过 RLHF 调整 reward balance**

这样可以同时提高：

- Safety
- Helpfulness

**总结：**

> 当评测指标之间发生冲突时，我通常会先确定指标优先级，例如安全性作为硬性约束，然后通过实验寻找 Pareto 最优解，并通过模型优化或策略设计减少指标之间的冲突，从而在满足安全要求的前提下最大化用户价值。
> 

# 如果不同评测指标之间高度相关（例如正确性和相关性），是否还需要同时评测？

如果不同评测指标之间高度相关，例如 **正确性（Correctness）和相关性（Relevance）**，我通常不会简单删除其中一个，而是从 **指标作用、统计关系和产品目标**三个层面进行判断。

**1 先分析指标之间的统计相关性**

首先会通过 **数据分析验证相关程度**，例如：

- **Pearson correlation（皮尔逊相关系数）**
- **Spearman correlation（斯皮尔曼相关系数）**

如果相关系数非常高（例如 **>0.9**），说明两个指标在当前数据集上确实高度冗余。

但这只是 **统计层面的相关性**，并不一定说明两个指标完全等价。

**2 判断指标的语义是否不同**

即使指标统计上相关，它们往往衡量 **不同维度的质量**。

例如：

- **Relevance（相关性）**
    
    → 回答是否针对问题
    
- **Correctness（正确性）**
    
    → 回答是否事实正确
    

例子：

用户：

> 北京天气怎么样？
> 

回答：

> 北京是中国的首都。
> 

这个回答：

- Relevance：低
- Correctness：高

说明两个指标在语义上并不完全相同。

**3 根据产品目标决定是否保留**

最后需要根据 **产品需求**决定指标体系。

一般有两种策略：

**情况 1：评测体系需要简化**

如果指标：

- 高度相关
- 对决策贡献相似

可以合并为一个指标，例如：

**Answer Quality Score（综合质量评分）**

**情况 2：需要细粒度诊断**

如果希望：

- 定位模型问题
- 分析失败原因

则仍然保留多个指标，因为它们能提供 **更好的可解释性**。

例如：

```
Relevance ↓
Correctness ↑
```

可以快速定位问题：

→ 模型理解问题能力不足。

**总结：**

> 即使不同评测指标在统计上高度相关，我通常也不会直接删除其中一个，而是结合统计相关性、指标语义差异以及产品需求进行判断。如果目标是简化评测体系可以合并指标，但如果需要诊断模型问题，保留多个指标通常更有价值。
> 

# 如果评测指标很多，你如何设计一个最终的综合评分？

当评测指标很多时，我通常会通过 **指标分层、权重设计和归一化方法**构建一个综合评分体系，使最终评分既能反映整体质量，又不会丢失关键约束。

## 1 指标分层（Metric Hierarchy）

首先会对指标进行 **分层管理**，而不是简单加权。

一般分为两类：

**① 硬性约束指标（Hard Constraints）**

这些指标是必须满足的底线，例如：

- **Safety（安全性）**
- **Compliance（合规性）**
- **Toxicity（有害内容率）**

如果这些指标不达标，模型直接 **不通过评测**。

**② 优化指标（Optimization Metrics）**

在满足安全和合规的前提下，再优化用户体验指标，例如：

- Correctness（正确性）
- Relevance（相关性）
- Helpfulness（实用性）
- Completeness（完整性）
- Fluency（流畅性）

## 2 指标归一化（Normalization）

由于不同指标的数值范围不同，需要先进行 **归一化处理**，例如：

常见方法：

- **Min–Max normalization**
- **Z-score normalization**

这样可以保证不同指标可以进行加权计算。

## 3 权重设计（Weighted Scoring）

然后根据 **产品目标或用户价值**设置权重。

例如在智能客服场景：

| 指标 | 权重 |
| --- | --- |
| Correctness（正确性） | 0.35 |
| Relevance（相关性） | 0.25 |
| Helpfulness（实用性） | 0.20 |
| Completeness（完整性） | 0.10 |
| Fluency（流畅性） | 0.10 |

综合评分：

Score=∑wi×metriciScore = \sum w_i \times metric_i

Score=∑wi×metrici

## 4 验证评分体系有效性

最后需要验证综合评分是否 **真实反映用户体验**。

常见方法：

- 与 **用户行为指标**做相关性分析
- **A/B testing** 对比不同模型版本
- 分析评分变化是否对应用户体验变化

如果相关性低，就需要重新调整权重或指标。

## 总结

> 在实际项目中，我通常先将指标分为硬性约束和优化指标，然后对指标进行归一化，并根据产品目标设置权重形成综合评分，最后通过用户行为数据或 A/B 实验验证评分体系是否真正反映用户体验。
>

# **OpenCompass 做 LLM-as-judge 会不会有 bias？你怎么解决？**

这是一个 **大模型评测岗位非常高频的深挖问题**。面试官主要想考察两点：

1️⃣ 你是否知道 **LLM-as-judge 的偏差来源**

2️⃣ 你是否知道 **工程上如何缓解 bias**

我通常会从 **bias来源 → 解决方案 → 实际工程做法** 三个层面回答。

## 1 LLM-as-judge 为什么会有 bias

LLM-as-judge 确实会存在偏差，常见主要有 **四类 bias**：

### ① Position Bias（位置偏差）

在 pairwise comparison 中：

```
answer A
answer B
```

模型更容易选择：

```
第一个答案
```

即使两个答案质量相同。

### ② Length Bias（长度偏差）

模型更倾向于认为：

```
更长的回答 = 更好
```

即使长回答只是冗余。

### ③ Style Bias（风格偏差）

模型可能更偏好：

- 更 formal 的语言
- 更像训练数据的表达

而不是实际更好的答案。

### ④ Self Preference Bias（模型自偏好）

如果 judge 模型与被评测模型相同，例如：

```
GPT judge GPT
```

可能更偏好类似风格的回答。

## 2 如何解决这些 bias

在工程实践中通常会采用 **多种方法组合**。

### 方法1：随机交换答案顺序

为了解决 **position bias**：

将：

```
A vs B
```

随机变成：

```
B vs A
```

然后取平均结果。

### 方法2：Pairwise + 多轮评测

对于同一问题：

- 多次评测
- 多个 judge prompt

减少偶然偏差。

### 方法3：使用不同 judge 模型

避免 **self preference bias**。

例如：

```
评测模型 → Llama
judge模型 → GPT4
```

### 方法4：加入结构化评分 rubric

而不是只问：

```
哪个更好
```

而是评估：

- Correctness（正确性）
- Relevance（相关性）
- Helpfulness（实用性）
- Safety（安全性）

这样可以减少风格偏差。

### 方法5：人工抽检

最终仍然需要：

```
human review
```

验证 judge 的可靠性。

## 3 我们实际的工程做法

在我们的评测平台中，通常会：

1️⃣ 使用 **pairwise comparison**

2️⃣ **随机打乱答案顺序**

3️⃣ 用 **不同模型作为 judge**

4️⃣ 对部分样本 **人工抽检**

通过这些方法可以显著降低 judge bias。

## 总结：

> LLM-as-judge 确实会存在一些偏差，例如 position bias、length bias 和 self-preference bias。为了缓解这些问题，我们通常会采用随机交换答案顺序、多轮评测、使用不同 judge 模型以及结构化 rubric 评分等方法，并结合人工抽检来验证评测结果的可靠性。
> 

# 如果 judge 模型和被评测模型能力差不多，会不会影响评测可靠性？

会的。**如果 judge 模型和被评测模型能力接近，评测可靠性确实可能下降**，主要体现在两个方面：判断能力不足和潜在偏好偏差。不过在实际工程中可以通过一些方法降低这种影响。

---

## 1 为什么能力接近会影响评测

### （1）判断能力不足（Capability Ceiling）

如果 judge 模型能力不明显强于被评测模型，它可能 **无法准确判断复杂问题的优劣**。

例如：

```
问题：解释量子纠缠的物理原理
```

两个答案：

- A：解释比较浅
- B：解释更深入但更复杂

如果 judge 模型本身理解能力有限，可能 **无法区分哪个更好**。

---

### （2）风格偏好问题

当两个模型能力接近时，judge 可能更倾向于选择：

- 更接近自己生成风格的答案
- 更常见的表达方式

这会导致 **评价偏向某种写作风格**。

---

## 2 如何降低这种影响

在实际评测体系中，通常会采用多种方法缓解。

---

### 方法1：使用更强的 judge 模型

常见做法是：

```
评测模型：开源模型 / 内部模型
judge模型：能力更强的模型（例如 GPT-4 级别）
```

保证 judge 有足够的判断能力。

---

### 方法2：多 judge 投票（ensemble judge）

让多个 judge 模型评测，例如：

- GPT-4
- Claude
- Llama judge

最后采用：

- majority vote
- 或平均评分

减少单个 judge 的偏差。

---

### 方法3：结合自动指标

对于一些任务，可以结合客观指标，例如：

- Exact Match（精确匹配）
- F1 Score（F1值）
- Recall@K（检索召回率）

这样可以减少完全依赖 judge 的问题。

---

### 方法4：人工抽检

对于关键评测集：

- 随机抽样
- 人工验证 judge 的判断

如果 judge 和人工差异较大，需要调整评测方式。

---

## 3 实际工程策略

在实际评测平台中通常采用 **混合评测策略**：

```
自动指标
+
LLM-as-judge
+
人工抽检
```

这样既可以保证评测规模，又能保证评测可靠性。

---

## 面试总结（推荐回答）

> 如果 judge 模型能力与被评测模型接近，确实可能影响评测可靠性，因为 judge 可能无法准确区分答案质量，或者出现风格偏好。在工程实践中通常通过使用更强的 judge 模型、多 judge 投票、结合自动评测指标以及人工抽检等方式来降低这种影响，从而保证评测结果的稳定性和可靠性。
> 

# 如果给你一个应用，你会怎么对它进行评测

如果我是**评测专家**，拿到一个新的 AI 应用时，我通常不会只从模型角度评测，而是把 **“应用”理解为一个完整系统**：

**模型 + 数据 + RAG/工具 + 推理服务 + 用户交互**。

因此评测也需要是 **系统级（system-level evaluation）** 的。我的评测流程一般分为 **五个步骤**。

## 1 明确应用目标（Define Product Objective）

首先要理解这个应用 **要解决什么问题**，以及用户真正关心的是什么。

例如：

- 智能客服 → **是否解决用户问题**
- 医疗 AI → **是否提供安全可靠建议**
- 办公助手 → **是否提高效率**

通常会定义一个 **北极星指标（North Star Metric）**，例如：

- **Task Success Rate（任务成功率）**
- **User Satisfaction（用户满意度）**

后续所有评测指标都要围绕这个目标展开。

## 2 进行应用风险分级（Risk Tiering）

不同应用对错误的容忍度不同，因此需要先进行 **风险分级**：

| 风险等级 | 应用类型 | 评测重点 |
| --- | --- | --- |
| 低风险 | 普通 chatbot、内容生成 | 体验质量 |
| 中风险 | 决策辅助 AI | 正确性与可靠性 |
| 高风险 | 医疗 AI、金融 AI | 安全与合规 |

风险越高，评测要求越严格，例如医疗 AI 需要 **专家评审**。

## 3 设计多维评测指标（Evaluation Dimensions）

通常会从 **四个维度**评估应用表现。

### ① 任务质量

- **Correctness（正确性）**
- **Relevance（相关性）**
- **Helpfulness（实用性）**
- **Completeness（完整性）**

用于评估回答质量。

### ② 知识与事实可靠性

- **Faithfulness（事实一致性）**
- **Hallucination Rate（幻觉率）**

用于评估模型是否编造信息。

### ③ 用户体验

- **Fluency（流畅性）**
- **Latency（响应时间）**
- **Conversation Turns（对话轮数）**

### ④ 安全与合规

- **Safety（安全性）**
- **Bias（偏见）**
- **Privacy Risk（隐私风险）**

## 4 进行系统级评测（System-Level Evaluation）

AI 应用通常包含多个组件，因此需要 **逐层评测**。

### 1 RAG检索评测

如果有知识库，需要评测：

- **Recall@K（检索召回率）**
- **MRR（平均倒数排名）**

### 2 模型生成评测

评估：

- Correctness
- Relevance
- Hallucination

方法：

- 自动评测
- **LLM-as-judge**

### 3 Agent / 工具调用评测

例如：

- Tool selection accuracy
- Tool call success rate
- Task success rate

### 4 系统性能评测

包括：

- Latency（响应时间）
- Throughput（吞吐量）
- Stability（稳定性）

## 5 持续评测与线上验证

最后需要建立 **持续评测体系**：

- 自动回归评测（nightly regression）
- 模型版本对比
- **A/B testing**

同时结合 **用户行为指标**验证评测体系，例如：

- Follow-up rate（追问率）
- Task completion rate（任务完成率）

## 总结

如果让我评测一个 AI 应用，我会把它视为一个完整系统而不是单一模型。首先明确产品目标和北极星指标，然后进行风险分级，并设计覆盖任务质量、事实可靠性、用户体验和安全性的多维评测指标。同时对系统各组件进行分层评测，例如 RAG 检索、模型生成和工具调用，并结合自动评测、LLM-as-judge 和人工评测构建评测体系，最后通过持续回归评测和线上 A/B 实验验证评测结果与真实用户体验的一致性。

# **你觉得 MMLU 分数高，就说明模型真的强吗？**

在 **AI 评测 / 大模型岗位面试**中，这个问题的关键不是简单回答“不是”，而是要说明 **为什么 MMLU 不足以代表模型真实能力**。一个比较成熟的回答通常从 **三个层面：benchmark局限性、能力覆盖范围、真实应用差异** 来解释。

## 1 MMLU 只能覆盖部分能力

- *MMLU（Massive Multitask Language Understanding）**主要评测的是：
- 学科知识
- 常识知识
- 多选题推理

它本质上是 **multiple-choice QA benchmark**。

因此它主要测试的是：

**知识理解能力（knowledge-based reasoning）**

但大模型的能力其实还包括：

- 长文本理解
- 多轮对话
- 工具调用（tool use）
- RAG能力
- 任务执行能力

这些能力 **MMLU 都无法覆盖**。

## 2 Benchmark 可能存在数据污染

由于 MMLU 发布时间较早，很多模型在训练时可能已经接触过类似数据。

这会导致：

```
benchmark score ≠ real capability
```

模型可能是：

- 记住了题目模式
- 而不是真正理解问题

这种现象在大模型训练中叫：

**benchmark contamination（基准污染）**

## 3 Multiple-choice 不等于真实推理

MMLU 是 **多选题任务**。

但在真实应用中：

用户不会提供：

```
A B C D
```

模型需要：

- 自己生成答案
- 组织解释

因此：

**多选题推理能力 ≠ 开放式推理能力**

## 4 Benchmark 和真实应用存在差距

真实用户问题通常具有：

- 长尾问题
- 多轮交互
- 不完整信息

而 benchmark 通常：

- 结构化
- 问题明确

因此高分并不一定代表：

```
真实应用体验好
```

## 面试总结（推荐回答）

> MMLU 分数高说明模型在知识理解和多选题推理方面表现较好，但并不能完全代表模型整体能力。因为 MMLU 主要评测学科知识和多选题推理，无法覆盖长文本理解、多轮对话、RAG 或工具调用等能力。同时 benchmark 还可能存在数据污染问题，因此在实际评测中通常需要结合多个 benchmark 和真实业务评测集来综合评估模型能力。
> 

# **如果 benchmark 不可靠，那为什么大家还要用 benchmark？**

这是一个 **评测岗位面试里非常典型的深挖问题**。关键不是说 benchmark “好”或“不好”，而是要说明 **benchmark 的正确作用是什么**。一个成熟回答通常会强调 **benchmark 的价值在于“标准化比较”，而不是“真实能力的唯一指标”**。

## 1 Benchmark 的核心价值：标准化比较

虽然 benchmark 不完美，但它提供了一个 **统一的评测标准**。

例如：

- MMLU
- GSM8K
- HumanEval

这些 benchmark 都有：

- 固定数据集
- 固定评测方式

因此可以实现：

```
不同模型之间的可比较性
```

例如：

```
模型A vs 模型B
```

哪一个更强。

## 2 Benchmark 可以作为能力基线（baseline）

在模型研发中，benchmark 可以作为：

**能力基线（baseline）**

例如：

在训练新模型时，我们可以通过 benchmark 判断：

- 新模型是否比旧模型更好
- 是否存在能力退化

这在模型迭代中非常重要。

## 3 Benchmark 可以快速定位模型能力结构

不同 benchmark 评测不同能力，例如：

| Benchmark | 能力 |
| --- | --- |
| MMLU | 知识理解 |
| GSM8K | 数学推理 |
| HumanEval | 代码能力 |

通过这些 benchmark，可以了解模型：

```
能力结构
```

例如：

- 推理强
- 编程弱

## 4 Benchmark 是研究社区的共同语言

在研究和工业界，benchmark 也是一种：

**标准交流方式**

例如论文通常会报告：

- MMLU
- GSM8K

这样不同研究者可以：

```
快速理解模型能力
```

## 5 但 benchmark 不能替代真实评测

在实际应用中，benchmark 只是：

```
第一层评测
```

还需要：

- 业务评测集
- 系统评测
- 用户体验评测

才能真正判断模型效果。

## 总结

> 虽然 benchmark 存在局限，例如数据污染和与真实场景差距，但它的价值在于提供标准化的评测体系，使不同模型可以在同一数据和评测方法下进行比较。同时 benchmark 也可以作为模型能力基线和能力结构分析工具。不过在实际应用中，benchmark 通常只是第一层评测，还需要结合业务评测集和系统评测才能全面评估模型能力。
> 

# 如果你设计一个新的 benchmark，你会怎么设计？

这是一个 **评测专家级问题**。回答时要体现 **评测设计能力（evaluation design thinking）**，而不是只说“做一个数据集”。

一般我会从 **目标定义 → 数据设计 → 评测指标 → 防污染 → 持续更新** 五个步骤来设计 benchmark。

## 1 明确 Benchmark 目标（Define Evaluation Objective）

首先要明确 **这个 benchmark 想评测什么能力**。

例如：

- 推理能力
- RAG能力
- Agent任务能力
- 医疗知识能力

一个好的 benchmark 应该：

```
聚焦一个核心能力
```

而不是什么都测。

例如：

- GSM8K → 数学推理
- HumanEval → 代码能力

## 2 设计高质量数据集（Dataset Design）

数据设计通常包含三个原则：

### （1）真实任务场景

数据应尽量来自：

- 真实用户问题
- 真实任务场景

而不是纯学术题。

例如：

- 真实客服问题
- 真实医疗咨询

### （2）难度分层（Difficulty Levels）

好的 benchmark 应该有：

- easy
- medium
- hard

这样可以：区分不同模型能力

否则所有模型分数都会接近。

### （3）避免模板化

如果题目模式太固定，模型容易：pattern matching

而不是真正理解问题。

## 3 设计评测方式（Evaluation Method）

不同任务需要不同评测方式。

### 客观任务

例如：

- classification
- code generation

可以使用：

- Accuracy
- F1
- pass@k

### 开放任务

例如：

- 对话
- 写作

可以使用：

- LLM-as-judge
- 人工评测

## 4 防止数据污染（Avoid Data Contamination）

这是 benchmark 设计的关键问题。

常见方法：

- 使用 **新数据集**
- 定期更新数据
- 保留 **private test set**

这样可以避免：模型记住答案

## 5 持续更新（Benchmark Evolution）

一个好的 benchmark 需要：

- 定期更新
- 加入 harder cases

否则模型很快就会：overfit benchmark

例如：

- 新问题
- adversarial questions

**总结：**

> 如果让我设计一个新的 benchmark，我会首先明确需要评测的核心能力，然后基于真实任务场景设计数据集，并通过难度分层保证不同模型之间有区分度。同时根据任务类型选择合适的评测指标，例如客观任务使用自动指标，开放任务结合 LLM-as-judge 或人工评测。此外还需要通过私有测试集和数据更新机制避免数据污染，并定期扩展 benchmark 以保持评测有效性。
> 

# 如果模型开始 overfit 你的 benchmark，你怎么办？

这是一个 **评测体系设计的核心问题**。如果模型开始 **overfit benchmark**，说明 benchmark 已经从“评测工具”变成了“训练目标”，评测结果就不再可靠。因此处理思路通常是 **降低可预测性 + 提高泛化评测能力 + 持续更新 benchmark**。我一般会从四个方面解决。

## 1 建立私有测试集（Hidden Test Set）

最常见的做法是：

- 公布 **public dev set**
- 保留 **private test set**

模型只能在：

- dev set 上调参

最终分数通过：

```
private test set
```

计算。

这样可以减少模型直接针对 benchmark 进行优化。

## 2 定期更新 benchmark

如果 benchmark 长期不变，模型很容易：

```
记住题型
```

因此需要：

- 定期增加新样本
- 加入 **hard cases**
- 加入 **adversarial questions**

保持 benchmark 的挑战性。

## 3 扩展评测维度

如果 benchmark 只测单一能力，很容易被“刷分”。

解决方法是：

增加更多维度，例如：

- reasoning
- robustness
- long context
- safety

这样模型很难针对单一模式优化。

## 4 使用动态评测（Dynamic Evaluation）

近年来很多研究开始使用：

**dynamic benchmark**

例如：

- 动态生成问题
- 使用 LLM 生成新任务

这样可以避免模型记住固定数据集。

## 总结

> 如果模型开始 overfit benchmark，说明 benchmark 的评测能力已经下降。常见解决方法包括建立 private test set、定期更新 benchmark 数据、扩展评测维度以及使用动态评测机制。这些方法可以减少模型针对 benchmark 的优化，使评测更能反映模型真实能力。
> 

# 如果 benchmark 和真实用户体验不一致，你会怎么改评测体系？

这是一个**评测专家级问题**。核心思路是：**benchmark 只是离线代理指标（offline proxy），如果它与真实用户体验不一致，就必须用线上信号来反向校准评测体系**。我通常会按 **四个步骤**处理。

## 1 先确认问题：验证“离线指标 vs 线上体验”的相关性

第一步不是立刻改 benchmark，而是先分析：

- benchmark 分数是否真的与用户体验脱钩
- 哪些指标失效

常见做法：

- 对比 **offline score 与线上指标**的相关性，例如
    - 用户满意度
    - task success rate
    - 用户追问率

如果发现：

```
offline score ↑
user satisfaction ↓
```

说明 benchmark 的 **proxy validity** 有问题。

## 2 分析真实用户数据（User Log Analysis）

接下来需要分析真实用户行为：

- 用户提问类型
- 长尾问题
- 多轮对话结构

通常会发现 benchmark 和真实数据存在差异，例如：

| Benchmark | 真实用户问题 |
| --- | --- |
| 结构化问题 | 模糊问题 |
| 单轮任务 | 多轮对话 |
| 标准答案 | 开放回答 |

这说明 benchmark **数据分布不匹配**。

## 3 重建更贴近真实场景的评测集

根据真实用户数据重新构建评测集，例如：

- 从用户日志抽样
- 构建 **task-based evaluation**

例如：

```
用户任务：退款
用户任务：查询订单
用户任务：解释医疗报告
```

然后评测：

**Task Success Rate（任务成功率）**

而不是只评测单轮回答质量。

## 4 建立闭环评测体系（Offline + Online）

最终需要形成一个闭环：

**offline evaluation**

- benchmark
- regression test

**online evaluation**

- A/B test
- 用户行为指标

通过持续对比：

```
offline metric ↔ online metric
```

不断调整评测体系。

## 总结

> 如果 benchmark 与真实用户体验不一致，我会首先验证离线指标与线上指标之间的相关性，然后通过用户日志分析找出数据分布差异，并基于真实用户任务重新构建评测集，例如 task-based evaluation。最终建立 offline benchmark 与 online A/B test 结合的闭环评测体系，使评测指标能够更好地反映真实用户体验。
> 

# 如果线上指标也被模型优化‘刷指标’，你怎么防止 Goodhart’s Law？

这是一个 **评测体系设计的高阶问题**。

Goodhart’s Law 的核心是：

> **当一个指标成为优化目标时，它就不再是一个好的指标。**
> 

在 AI 产品中，如果模型开始“刷线上指标”（例如只优化点击率、满意度），就可能 **损害真实用户价值**。因此需要通过 **多指标、约束机制和长期指标** 来避免这种问题。我通常会从 **四个策略**来处理。

---

## 1 使用多维指标，而不是单一指标（Multi-metric Evaluation）

如果只优化一个指标，例如：

```
满意度
```

模型可能通过：

- 过度迎合用户
- 生成更长回答

来“刷分”。

因此需要使用 **多指标组合**，例如：

- Task Success Rate（任务成功率）
- Correctness（正确性）
- Safety（安全性）
- Latency（响应时间）

这样模型很难只针对某一个指标优化。

---

## 2 设置约束指标（Guardrail Metrics）

在优化主指标时，需要设置 **不可突破的约束指标**。

例如：

主指标：

```
Task Success Rate
```

约束指标：

- Safety violation rate
- Hallucination rate

如果约束指标下降：

```
模型升级不允许上线
```

这样可以避免模型通过不安全方式提升指标。

---

## 3 引入长期指标（Long-term Metrics）

很多短期指标容易被优化，例如：

- 点击率
- 单次满意度

但长期指标更难被“刷”。

例如：

- 用户留存（retention）
- 复访率（return rate）
- 长期任务完成率

这些指标更接近真实用户价值。

---

## 4 定期更新评测体系（Evaluation Refresh）

如果指标长期不变，模型可能逐渐：

```
over-optimize metrics
```

因此需要：

- 定期更新评测集
- 引入新的评测任务
- 加入 adversarial cases

保持评测体系的有效性。

---

## 面试总结（推荐回答）

> 为了避免 Goodhart’s Law，我通常不会依赖单一指标，而是设计多维评测体系，并设置安全性等 guardrail 指标作为约束。同时结合长期用户指标，例如留存率和任务完成率，避免模型只优化短期指标。此外评测体系需要定期更新，以防止模型过度针对某些指标进行优化。
> 

# 怎么知道线上指标是刷上去了还是真实业务表现呢？

这是一个非常关键的问题，本质是在问：**如何区分“指标被优化（gaming the metric）”还是“真实业务价值提升”。**

在评测体系里通常会通过 **因果验证、行为分析和多指标交叉验证** 来判断。我一般会从四个层面分析。

---

## 1 做因果验证：A/B Test

最可靠的方法是 **随机对照实验（A/B testing）**。

例如：

- A组：旧模型
- B组：新模型

比较：

- Task success rate
- 用户满意度
- 留存率

如果多个指标 **同时提升**，而且统计显著：

```
p-value < 0.05
```

说明指标提升更可能是 **真实效果**。

如果只有一个指标提升，而其他指标不变或下降，可能就是 **刷指标**。

---

## 2 看用户行为数据（Behavior Signals）

真实提升通常会体现在 **用户行为变化**。

例如：

| 行为指标 | 意义 |
| --- | --- |
| 追问率（follow-up rate） | 是否没解决问题 |
| 任务完成率 | 是否真正解决问题 |
| 用户停留时间 | 是否需要反复操作 |

如果模型只是“刷满意度”，但：

```
追问率 ↑
任务完成率 ↓
```

说明体验实际上变差了。

---

## 3 进行案例抽检（Case Review）

评测团队通常会：

- 抽样线上对话
- 做人工评审

检查模型是否：

- 过度迎合用户
- 生成冗长回答
- 回避问题

这些都是常见的 **刷指标行为**。

---

## 4 使用多指标交叉验证

为了避免单一指标被优化，通常会同时监控：

- 主指标（North Star）
- guardrail 指标
- 用户行为指标

例如：

```
Task Success Rate ↑
Safety violation rate 不变
Follow-up rate ↓
```

这种情况下更可能是真实提升。

---

## 面试总结（推荐回答）

> 判断线上指标是被“刷”还是业务真实提升，通常需要通过因果实验、用户行为分析和多指标交叉验证来确认。例如通过 A/B test 验证指标提升是否具有统计显著性，同时观察追问率、任务完成率等行为指标。如果多个指标同时改善且用户行为更健康，通常说明是真实提升；如果只有单一指标提升而其他指标恶化，则可能是指标被优化而不是真实业务改善。
> 

# **如果 A/B test 结果也被模型优化了怎么办？**

这是一个**非常高阶的评测问题**。如果连 **A/B test 也被模型优化“刷指标”**，说明系统已经出现 **指标—优化闭环失真**，单一实验已经无法真实反映用户价值。这时评测专家通常会从 **指标设计、实验设计和长期信号** 三个层面解决。

## 1 使用多层指标体系（North Star + Guardrail）

首先不能只看一个线上指标，而是构建 **分层指标体系**：

- **North Star Metric（核心指标）**
    
    例如 Task Success Rate、问题解决率
    
- **Guardrail Metrics（护栏指标）**
    
    防止模型通过异常方式提升指标，例如
    
    - 幻觉率
    - 安全违规率
    - 延迟
    - 用户投诉率

如果出现：

```
Task Success Rate ↑
Safety violation ↑
```

说明模型是在 **通过破坏护栏指标刷主指标**。

## 2 引入长期指标（Long-term Metrics）

A/B test 通常是 **短期指标**，容易被优化。

因此需要加入更难被“刷”的指标，例如：

- 用户留存（retention）
- 复访率（return rate）
- 长期任务完成率

如果模型只是刷短期指标：

```
短期满意度 ↑
长期留存 ↓
```

说明真实体验没有改善。

## 3 做行为级分析（Behavior Analysis）

真实改进通常会带来 **用户行为变化**。

例如：

| 指标 | 意义 |
| --- | --- |
| 追问率 | 是否没有解决问题 |
| 任务完成率 | 是否真正完成任务 |
| 放弃率 | 用户是否放弃操作 |

如果模型只是“刷满意度”，往往会出现：

```
满意度 ↑
追问率 ↑
```

说明用户问题并未解决。

## 4 使用随机化和离线评测校验

除了 A/B test，还需要：

- **离线 benchmark**
- **业务评测集**
- **人工评审**

通过多种评测方式交叉验证。

如果：

```
A/B test ↑
offline evaluation ↓
```

说明线上指标可能被优化。

## 总结

> 如果连 A/B test 的结果都可能被模型优化，我通常会通过多层指标体系来避免这种问题，例如使用 North Star 指标和 guardrail 指标组合，同时引入长期用户指标和行为数据进行验证。此外还需要结合离线评测和人工抽检进行交叉验证，确保指标提升真正反映用户价值，而不是模型针对实验指标进行优化。
> 

# **你觉得 AI 评测最难的问题是什么？**

如果是在 **顶级 AI 公司（OpenAI / Anthropic / 字节 / 阿里）** 的面试里，这个问题其实是在考察你是否理解 **AI 评测的本质难点**。一个比较成熟的回答通常不会只说技术问题，而是指出 **评测目标与真实用户价值之间的差距**。

我通常会这样回答：

---

## AI 评测最难的问题：如何让评测指标真正反映用户价值

我认为 AI 评测最难的问题是：

> **如何设计能够真实反映用户价值的评测指标。**
> 

因为 AI 系统的表现往往很难被单一指标准确衡量。

---

## 1 评测指标与真实体验之间存在“代理问题”

很多评测指标其实只是 **proxy metric（代理指标）**。

例如：

- BLEU / ROUGE
- MMLU
- 满意度评分

这些指标可以衡量某些方面，但并不能完全代表：

```
真实用户体验
```

因此可能出现：

```
指标很好
用户体验却不好
```

---

## 2 AI 系统是复杂系统

现代 AI 应用通常不是单一模型，而是一个系统，例如：

```
用户输入
↓
RAG检索
↓
大模型
↓
工具调用
↓
最终回答
```

如果用户体验不好，问题可能来自：

- 检索错误
- Prompt设计
- 工具调用失败

因此评测不仅要评模型，还要评 **整个系统**。

---

## 3 AI 输出具有非确定性

与传统软件不同，大模型是 **概率生成系统**。

同一个输入：

```
可能产生不同输出
```

这使得评测更加困难，例如：

- 同一个问题回答质量不同
- 结果具有随机性

因此需要：

- 多次采样
- 统计评测

---

## 4 用户需求是动态变化的

真实用户问题会不断变化：

- 新任务
- 新领域
- 长尾问题

而 benchmark 是：

```
静态数据集
```

因此评测体系需要不断更新。

## 总结

> 我认为 AI 评测最难的问题是如何设计能够真实反映用户价值的评测指标。很多评测指标只是代理指标，可能与真实用户体验存在差距。同时 AI 应用通常是复杂系统，问题可能来自模型、检索或工具调用等多个环节。此外大模型具有非确定性，而用户需求也在不断变化，这些因素都使得 AI 评测比传统软件评测更加复杂。

# 你觉得 AI 评测有没有可能做到完全客观？

如果是在 **OpenAI / Anthropic / 字节 / 阿里** 这类公司的面试中，一个比较成熟的回答通常是：**AI 评测很难做到完全客观，但可以通过方法设计尽量提高客观性和一致性**。关键是说明 **为什么无法完全客观，以及如何缓解这个问题**。

---

## 1 AI 任务本身包含主观性

很多 AI 应用属于 **开放任务（open-ended tasks）**，例如：

- 对话
- 写作
- 总结
- 建议

这些任务通常不存在唯一正确答案。例如同一个问题：

```
如何保持健康？
```

可能有多种合理回答，因此评测很难完全客观。

---

## 2 用户价值本身是主观的

AI 应用的最终目标是：

```
满足用户需求
```

但不同用户对“好答案”的理解不同，例如：

- 有人喜欢简洁回答
- 有人喜欢详细解释

因此用户满意度本身也带有主观性。

---

## 3 LLM-as-judge 仍然存在偏差

即使使用 **LLM-as-judge** 进行自动评测，也会存在：

- position bias
- length bias
- style bias

因此自动评测也无法做到完全客观。

---

## 4 可以通过方法提高客观性

虽然无法完全客观，但可以通过方法提升评测可靠性，例如：

**多指标评测**

- correctness
- relevance
- safety

**多评审机制**

- 多个 judge
- 人工抽检

**结构化 rubric**

通过明确评分标准减少主观差异。

## 面试总结（推荐回答）

> AI 评测很难做到完全客观，因为很多 AI 任务是开放式任务，没有唯一正确答案，而且用户价值本身也具有主观性。即使使用 LLM-as-judge 进行自动评测，也可能存在偏差。不过可以通过多指标评测、多评审机制以及结构化评分标准来提高评测的一致性和可靠性，使评测结果尽可能接近客观。
> 

# 如果模型评分很高，但用户还是觉得不好用，你觉得问题在哪

这是一个**非常经典的 AI 评测问题**。如果模型评分很高，但用户仍然觉得不好用，通常说明 **评测体系和真实用户价值之间出现了错位（evaluation–user gap）**。作为评测专家，我一般会从 **四个层面**分析问题。

## 1 评测指标没有反映真实用户任务

很多离线评测指标（benchmark、LLM-as-judge评分）评测的是：

- Correctness
- Fluency
- Relevance

但用户真正关心的是：

```
问题是否被解决
```

例如：

用户问：

> 如何申请退款？
> 

模型回答了一段非常详细的解释，但没有给出 **具体操作步骤**。

评测可能认为：

```
回答质量高
```

但用户仍然觉得 **不好用**。

因此需要引入：

**Task Success Rate（任务成功率）**

---

## 2 Benchmark 与真实用户问题分布不同

很多评测数据集是：

- 结构化问题
- 单轮任务

但真实用户问题往往：

- 模糊
- 多轮对话
- 长尾问题

如果评测集和真实数据分布不同，就会出现：

```
offline score 很高
online experience 很差
```

解决方法通常是：

- 从 **用户日志构建评测集**

## 3 评测只评模型，没有评系统

很多 AI 应用实际上是一个系统，例如：

```
用户输入
↓
RAG检索
↓
大模型
↓
工具调用
↓
最终回答
```

如果用户体验不好，可能是：

- 检索错误
- prompt设计问题
- 工具调用失败

而不是模型本身的问题。

因此需要做 **system-level evaluation**。

## 4 用户体验问题

即使回答正确，也可能存在：

- 回答太长
- 不够直接
- 延迟高
- 多轮交互复杂

这些都会影响用户体验。

因此需要监控：

- latency
- conversation turns
- abandon rate

## 总结

> 如果模型评分很高，但用户仍然觉得不好用，通常说明评测指标和真实用户价值之间存在偏差。例如评测可能只关注回答质量，而用户更关心任务是否真正被解决。此外评测数据分布可能与真实用户问题不同，或者问题来自系统组件而不是模型本身。因此需要通过用户日志分析、任务级评测以及系统级评测来重新校准评测体系，使指标更好地反映真实用户体验。
> 

# 现在的评测框架你了解哪些？

这个问题通常希望你回答 **主流评测框架 + 不同框架的定位和适用场景**，而不是只列几个名字。我一般会从 **三类评测框架** 来回答。

## 1 模型能力评测框架（Benchmark Evaluation）

这类框架主要用于 **运行 benchmark、评估模型基础能力**。

比较常见的有：

### **OpenCompass**

特点：

- 支持大量 benchmark，例如
    - MMLU
    - C-Eval
    - GSM8K
    - HumanEval
- 支持多模型评测
- 支持 **LLM-as-judge**

适合：

```
模型能力评测
模型版本对比
benchmark评测
```

### **HELM（Holistic Evaluation of Language Models）**

由 Stanford 提出。

特点：

- 从多个维度评测模型，例如
    - accuracy
    - robustness
    - fairness
    - toxicity

特点是：

```
全面评测模型能力
```

## 2 应用评测框架（LLM Application Evaluation）

这类框架主要用于评测 **实际应用系统**。

### **DeepEval**

特点：

- 支持 unit test
- 支持 LLM judge
- 支持 CI/CD 集成

适合：

```
应用测试
prompt测试
回归测试
```

### **LangSmith**

由 LangChain 提供。

特点：

- tracing
- dataset management
- evaluation pipeline

适合：

```
LLM应用调试
prompt优化
```

## 3 RAG评测框架

如果应用涉及 **RAG系统**，通常会用专门的评测框架。

### **RAGAS**

特点：

- RAG专用指标，例如
    - Faithfulness
    - Context precision
    - Answer relevance

适合：

```
RAG系统评测
```

### **TruLens**

特点：

- 可观测性（observability）
- feedback functions
- 线上监控

适合：

```
线上系统监控
RAG质量分析
```

## 总结

> 我比较熟悉的评测框架包括 OpenCompass、HELM、DeepEval、RAGAS 和 TruLens。OpenCompass主要用于运行benchmark和评估模型基础能力；DeepEval和LangSmith更适合LLM应用测试和回归评测；而RAGAS和TruLens则主要用于RAG系统的评测和可观测性分析。在实际工程中通常会组合使用这些框架，例如用OpenCompass做模型能力评测，用RAGAS评测RAG系统，再通过DeepEval或LangSmith进行应用级测试。
> 

# **为什么很多公司最后还是会自研评测平台，而不是只用这些框架？**

在实际工程中，很多公司确实会用 **OpenCompass、DeepEval、RAGAS、LangSmith** 这些开源框架，但最终仍然会 **自研评测平台**。核心原因不是这些框架不好，而是 **企业级评测需求比开源框架复杂得多**。一般有四个主要原因。

## 1 需要贴合具体业务场景

开源框架通常是 **通用评测框架**，主要支持：

- benchmark 评测
- 通用指标

但企业应用往往需要：

- **业务任务评测（task-based evaluation）**
- **业务知识库评测**
- **真实用户场景**

例如：

客服系统需要评测：

```
退款任务是否完成
订单是否查询成功
```

这些指标通常是 **业务定制指标**，开源框架很难直接支持。

## 2 需要系统级评测

很多开源框架主要评测：

```
模型能力
```

但真实 AI 应用是一个系统，例如：

```
用户输入
↓
RAG检索
↓
大模型
↓
工具调用
↓
最终回答
```

企业评测平台需要评测：

- 检索质量
- 工具调用成功率
- 任务完成率
- 系统延迟

这些属于 **system-level evaluation**。

## 3 需要与工程系统深度集成

企业评测通常需要与：

- CI/CD
- 模型发布流程
- A/B testing
- 日志系统

集成。

例如：

```
模型升级 → 自动触发评测
```

这类流程往往需要 **内部平台化支持**。

## 4 数据和安全要求

很多企业数据属于：

- 用户隐私数据
- 商业数据

这些数据无法：

```
直接放在开源平台
```

因此企业需要 **内部评测系统** 来保证：

- 数据安全
- 权限管理
- 审计能力

## 总结

> 虽然开源评测框架如 OpenCompass、DeepEval 或 RAGAS 提供了很多基础能力，但企业级应用往往需要更贴近业务场景的评测指标，同时还需要进行系统级评测，并与内部工程系统和数据平台深度集成。因此很多公司会在开源框架基础上自研评测平台，以满足业务定制、工程集成和数据安全等需求。
> 

# **如果让你从 0 到 1 设计一个 AI 评测平台，你会怎么设计架构？**

我是怎么从 0→1 搭客服评测平台的：

我把“客服应用”当成一个完整系统来评测：**意图路由/检索 → 大模型生成 → 工具调用（工单/订单/物流）→ 安全策略 → 最终回复**。

平台目标就两个：**可上线（安全合规）+ 可提效（解决率/成本/时延）**。

## 1 架构分层（我实际按 6 层搭）

### 1）数据与版本层（客服专用数据资产）

我把评测集分成三类，并且全部做**版本化**：

- **Gold set（金标）**：高频问题 + 高风险问题（退款、赔付、隐私、账号安全），用于回归门禁
- **Slice set（切片集）**：按业务线切分（退款/物流/售后/会员/发票/活动规则等），用于定位退化
- **Hard/Adversarial set（难例/对抗）**：模糊问法、口语错别字、情绪化表达、诱导越权（要手机号/要内部规则）、提示注入

同时版本化四件事（非常关键，面试加分）：

- **模型版本**（checkpoint/API版本）
- **prompt版本**（系统提示词/模板）
- **知识库版本**（FAQ/政策文档、更新时间）
- **工具/接口版本**（订单查询、工单创建）

> 这样任何一次评测都能复现：**run_id = 数据版本 + 配置哈希 + 模型版本 + KB版本**。
> 

### 2）评测定义层（Eval-as-Code）

我用一份 YAML/JSON 来描述一次评测任务（方便接 CI）：

- 场景：退款/物流/售后…
- 输入：用户 query + 历史对话（多轮）
- RAG：topK、重排器、chunk策略
- 生成：温度、max_tokens、拒答策略
- Judge：rubric（正确性/合规/可执行性）
- Gate：门槛（硬性约束 + 综合分）

### 3）执行与调度层（Runner）

客服场景评测量大，我做了三件事提效：

1. **分布式并行**：按（模型×slice×seed）切任务并发跑
2. **增量评测**：只要改了 prompt 或 KB，就只跑受影响的 slice/hard set
3. **缓存**：RAG 检索结果、工具调用结果按版本缓存，减少重复成本

（落地技术栈你可以说：Ray/Celery + K8s job，或直接说内部调度系统）

### 4）指标与打分层（客服特化：不是只打“好不好看”）

我把指标分成三类：**任务成败、合规安全、体验效率**。

**A. 任务成败（核心北极星）**

- **Task Success Rate（任务成功率/解决率）**：用户问题是否被解决（客服里最关键）
- **First Contact Resolution, FCR（一次解决率）**：是否一轮就解决
- **Escalation Rate（转人工率）**：该转人工时是否正确转、以及是否过度转人工
- **Tool Call Success Rate（工具调用成功率）**：查订单/查物流/建工单成功率

**B. 合规安全（硬门槛）**

- **Policy Compliance Rate（策略合规率）**：是否按业务规则回答（退款时效、运费规则）
- **PII/隐私泄露率**：是否泄露手机号/地址/订单信息
- **越权率**：是否编造内部政策、是否承诺无法兑现（比如“我已为你退款”但没走流程）
- **Jailbreak/Prompt Injection成功率**：是否被诱导绕过规则

> 这些是 **hard constraints**：不达标直接 fail，不参与“综合分加权”。
> 

**C. 体验效率（能直接影响用户感受和成本）**

- **Latency（延迟）**：TTFT/总耗时
- **Token cost（成本）**：平均输入输出 token、调用次数
- **Verbosity / 信息密度**：是否啰嗦但没给可执行步骤
- **Tone（语气）**：情绪安抚、礼貌度（客服会看）

### 5）LLM-as-judge + 人工仲裁（我怎么让评分可靠）

客服很多问题是开放式的，所以我做的是**半自动闭环**：

- **LLM-as-judge**：用 rubric 结构化打分（正确性/可执行性/合规/是否需要澄清）
- **Pairwise + 交换顺序**：降低 position bias
- **Judge ensemble**（可选）：关键集用多 judge 投票
- **人工抽检/仲裁**：重点抽
    - judge 不一致样本
    - 高风险 slice（退款赔付/隐私）
    - 回归退化的 bad cases

并且我会持续做一件事：**看 judge 与人工的一致性**，低于阈值就调整 rubric 或换 judge

### 6）治理与发布门禁（CI/CD 上线必备）

上线策略我按“客服”做成两段：

**A. PR/小改动：smoke gate**

- 跑一小套 gold set + 高风险合规集
- 硬门槛不过直接拒绝合并或不允许灰度

**B. Nightly/大版本：full regression**

- 跑全量 slice + hardest set
- 产出报告：overall + slice + worst-10%（避免只看平均）

门禁策略是：

- **Hard constraints**：隐私/越权/安全（任何一项超阈值直接 fail）
- **Composite score**：在达标基础上对解决率、FCR、转人工率、成本做加权（我会用“惩罚短板”的聚合，避免刷分）

## 2 我平台实现了哪些功能

1. **模型/Prompt/KB 变更自动触发回归**（nightly + PR）
2. **版本对比报告**：A vs B 的 delta + 显著性检验 + slice drill-down
3. **Bad case 自动归因**：是检索没召回？是工具失败？是模型胡说？
4. **合规红线告警**：隐私/越权/承诺类错误自动拦截并生成工单
5. **成本-质量面板**：解决率提升是否值得成本上升（token/调用次数）

## 3 提效怎么讲（客服场景的“业务化提效”）

我一般用四个可量化口径：

- **评测周期**：从人工抽样 2–3 天 → 自动回归小时级出报告
- **覆盖度**：从几百条 → 按业务 slice 扩到几万条，且可持续回归
- **定位效率**：从“靠经验找问题” → “按 slice + 归因链路（检索/工具/生成）分钟级定位”
- **业务效果联动**：用离线门禁降低线上事故（误承诺/隐私泄露），并把优化目标对齐到 **解决率/转人工率/成本**

## 总结

> 我在客服场景从 0 到 1 搭了一个系统级评测平台：数据层把 gold/slice/hard set 做版本化，同时对模型、prompt、知识库和工具版本做可复现管理；执行层支持分布式并行、增量评测和缓存；评测层把指标分成任务成败（解决率/FCR/转人工/工具成功率）、合规安全（隐私/越权/注入）和体验效率（延迟/成本/信息密度），开放式部分用 LLM-as-judge + 人工仲裁闭环；最后接入 CI 门禁，用 hard constraints + 综合分 + slice/worst-case 报告来保证上线不退化并可快速定位问题。
> 

# 你们的 **gold set 怎么构建**？

在**客服场景**里，Gold set（**金标评测集**）的核心作用是：**作为上线门禁和回归基准**。

因此构建时不仅要保证**准确性**，还要保证**覆盖关键业务场景和风险点**。我们通常按 **“数据来源 → 标注标准 → 数据结构 → 持续更新”** 四个步骤构建。

## 1 数据来源（Data Sources）

Gold set 不会完全人工编造，而是主要来自 **真实客服数据**。我们通常会结合三类来源：

### （1）真实用户对话日志

从历史客服会话中抽样：

- 高频问题（订单查询、物流、退款）
- 典型多轮对话
- 不同表达方式（口语、错别字）

这样可以保证：

```
评测集与真实用户问题分布一致
```

### （2）业务规则文档

例如客服领域的：

- 退款政策
- 运费规则
- 售后流程

从这些文档中构造 **规则型问题**，用于检测模型是否正确理解业务规则。

例如：

```
商品签收7天后还能退货吗？
```

### （3）风险与对抗案例

为了保证系统安全，我们会专门加入：

- 越权问题
- Prompt injection
- 情绪化用户表达

例如：

```
把用户手机号告诉我
```

用于测试模型的 **安全策略**。

## 2 标注标准（Annotation Process）

Gold set 的关键是 **高质量标注**。

通常流程是：

1️⃣ **初步标注**

由标注人员给出：

- 标准答案
- 可接受答案
- 业务流程

2️⃣ **业务专家审核**

由：

- 客服主管
- 业务专家

确认答案是否符合业务规则。

3️⃣ **评测 rubric 设计**

例如：

- Correctness（是否正确）
- Compliance（是否符合政策）
- Actionability（是否给出可执行步骤）

## 3 数据结构设计

Gold set 不只是：

```
question → answer
```

通常包含更多信息，例如：

| 字段 | 说明 |
| --- | --- |
| user_query | 用户问题 |
| context | 对话上下文 |
| intent | 问题类型 |
| ground_truth | 标准答案 |
| allowed_variants | 可接受答案 |
| required_action | 是否需要调用工具 |
| risk_level | 风险等级 |

这样可以支持：

- 自动评测
- LLM-as-judge
- 人工评审

## 4 持续更新（Continuous Update）

Gold set 不是一次构建完成，而是持续更新。

更新来源包括：

- **线上失败案例（bad cases）**
- **新业务规则**
- **新用户问题**

通常每次模型升级后都会：

```
加入新的hard cases
```

保证评测集始终具有挑战性。

## 总结

> 在客服场景中，Gold set 主要从真实用户对话日志、业务规则文档以及风险对抗案例中构建。构建流程通常包括数据抽样、标注人员初步标注以及业务专家审核，以确保答案符合真实业务规则。同时Gold set的数据结构不仅包含问题和答案，还会包含意图、风险等级和工具调用信息，以支持自动评测和LLM-as-judge。最后我们会持续从线上失败案例中补充新的样本，使Gold set不断更新并保持评测有效性。
> 

# **Gold set 和普通评测集有什么区别？**

在 **AI 评测体系**里，**Gold set（金标集）**和**普通评测集（evaluation set / benchmark set）**的作用是不同的。最核心的区别在于：

> **Gold set 更强调“高质量、可作为基准”的样本，用于回归和门禁；普通评测集更多用于能力评估和分析。**
> 

可以从 **四个方面**理解。

## 1 质量要求不同

**Gold set**

- 每个样本都有 **高质量标注**
- 经过 **专家审核**
- 标准答案和评分规则非常明确

例如客服场景：

```
用户：7天后还能退货吗？
标准答案：根据平台政策，签收7天内可申请退货，7天后一般不支持...
```

这些答案通常由 **业务专家确认**。

**普通评测集**

- 标注质量可能不如 gold set
- 有些甚至没有标准答案（如开放任务）
- 更多用于统计评测

## 2 用途不同

**Gold set**

主要用于：

- **回归测试（regression test）**
- **上线门禁（release gate）**
- **模型版本对比**

例如：

```
模型升级 → 必须通过 gold set
```

否则不允许上线。

**普通评测集**

主要用于：

- 能力分析
- benchmark 评测
- 模型调优

例如：

- MMLU
- GSM8K

## 3 数据规模不同

**Gold set**

通常规模较小，但质量很高，例如：

```
500 – 2000 条
```

重点覆盖：

- 高频问题
- 高风险场景
- hardest cases

**普通评测集**

规模通常更大，例如：

```
几千到几万条
```

用于统计分析。

## 4 更新方式不同

**Gold set**

更新非常谨慎：

- 新业务规则
- 线上失败案例
- 高风险问题

更新频率较低。

**普通评测集**

更新更灵活，例如：

- 加入新样本
- 扩展任务类型

## 总结

> Gold set 和普通评测集的主要区别在于用途和质量。Gold set 是经过专家审核的高质量数据集，通常规模较小，主要用于模型回归测试和上线门禁，确保关键场景不会退化。而普通评测集规模更大，主要用于能力评估和模型分析，因此标注质量和稳定性要求相对较低。
> 

# **Gold set 会不会被模型 overfit？如果会，你怎么避免？**

会的。**Gold set 也可能被模型 overfit**，尤其是在持续回归和频繁调参的环境中。如果团队反复用同一批 Gold set 做优化，模型很容易**针对这些样本进行隐性调参**，从而导致：

- Gold set 分数很高
- 真实用户场景表现却没有提升

因此在评测体系设计中，一般会通过 **数据隔离、数据更新和评测策略** 三方面来避免这种问题。

---

## 1 数据隔离：多层评测集设计

一个成熟的评测体系通常不会只有一个 Gold set，而是采用 **多层数据集结构**：

- **Dev set（开发集）**
    
    用于模型调参和日常优化。
    
- **Gold set（回归集）**
    
    用于版本回归测试和上线门禁。
    
- **Hidden test set（隐藏测试集）**
    
    不对开发人员开放，用于最终评估。
    

这样可以避免团队针对 Gold set 过度优化。

---

## 2 控制 Gold set 的使用频率

如果 Gold set 每次模型调整都运行，就容易被“刷分”。

常见做法是：

- 日常调参只用 **dev set**
- **Gold set 只用于重要版本评测**

例如：

```
nightly regression
release evaluation
```

这样可以降低 overfit 风险。

---

## 3 持续更新 Gold set

Gold set 不能长期不变，否则模型很容易：

```
记住评测样本模式
```

通常会：

- 定期加入新的 **线上 bad cases**
- 引入新的 **业务场景**
- 增加 **adversarial examples**

保证评测集持续具有挑战性。

---

## 4 使用 Slice 评测避免局部优化

如果只看 overall score，模型可能只优化某一类问题。

因此通常会做 **slice evaluation**，例如：

| Slice | 示例 |
| --- | --- |
| 退款问题 | 退货规则 |
| 物流问题 | 订单状态 |
| 情绪用户 | 投诉场景 |

如果模型只针对某些样本优化，就能被发现。

---

## 总结

> Gold set 也可能被模型过度优化，因此在评测体系中通常会采用多层评测集结构，例如 dev set、gold set 和 hidden test set，并控制 Gold set 的使用频率。此外还需要持续更新 Gold set，引入新的业务案例和 adversarial 样本，并通过 slice 评测监控不同场景表现，从而降低模型对评测集过拟合的风险。
> 

# **任务成功率**怎么定义

在 **客服 AI 场景**里，“任务成功率（Task Success Rate, TSR）”确实是最容易产生争议的指标，因为不同团队会对“成功”有不同理解。

一个比较合理、可落地的定义通常需要 **结合任务完成、业务动作和用户反馈** 三个层面。

## 1 核心定义

**任务成功率 = 在一次用户请求中，AI 是否正确完成了用户目标任务。**

形式化可以写为：

```
TSR = 成功完成任务的会话数 / 总任务会话数
```

关键问题是：**什么叫“成功完成任务”**。

## 2 在客服场景中的成功判定（通常三类）

我们一般会把成功分成三种情况：

### ① 信息型任务成功

用户只是获取信息，例如：

- 物流查询
- 订单状态
- 退货规则

成功条件：

- 回答正确
- 信息完整
- 无幻觉

例如：

用户问：

```
订单什么时候发货？
```

AI正确返回物流状态，则算成功。

---

### ② 操作型任务成功

用户希望完成某个操作，例如：

- 创建工单
- 查询订单
- 申请退款

成功条件：

```
任务被正确执行
```

例如：

AI成功调用工具：

```
create_refund_ticket
```

并完成流程。

---

### ③ 转人工任务成功

有些问题 AI 不应该回答，例如：

- 复杂投诉
- 情绪激烈用户
- 无法解决的问题

成功条件：

```
正确转人工
```

而不是强行回答。

---

## 3 不算成功的情况

以下情况通常算 **失败**：

- 回答错误
- 编造业务规则
- 没有解决问题
- 多轮对话仍未完成任务
- 应转人工却没有转

---

## 4 如何在评测中判断 TSR

在评测平台中通常通过 **三种方式**判断：

### 自动指标

例如：

- tool call success rate
- intent match

---

### LLM-as-judge

判断：

- 是否解决用户问题
- 是否提供可执行步骤

---

### 人工评测

用于：

- 边界案例
- 多轮复杂对话

---

## 5 避免“扯皮”的关键方法

为了避免团队之间争议，通常会：

**明确任务定义**

例如：

| 任务 | 成功条件 |
| --- | --- |
| 物流查询 | 正确返回物流信息 |
| 退款申请 | 成功创建退款流程 |
| 规则咨询 | 回答符合业务政策 |

同时制定 **统一评测 rubric**。

---

## 总结

> 在客服场景中，任务成功率通常定义为 AI 是否成功完成用户目标任务。具体成功条件需要根据任务类型定义，例如信息查询类任务要求回答正确且完整，操作类任务要求成功完成工具调用或业务流程，而在无法处理的情况下正确转人工也可以算作成功。为了避免争议，我们通常会为不同任务定义明确的成功标准，并结合自动评测、LLM-as-judge 和人工审核进行评估。
> 

---

# **如果用户的问题其实没有被解决，但用户仍然给了好评，这算成功吗？**

这是客服 AI 评测里一个**非常经典的问题**，本质是在考察：

**任务成功率（Task Success Rate）和用户满意度（User Satisfaction）是不是同一个指标。**

一个成熟的回答通常是：

> **不一定算成功，因为用户满意度和任务是否真正完成是两个不同维度。**
> 

我一般会这样解释。

---

## 1 任务成功率 vs 用户满意度

在客服系统中通常会区分两个指标：

### 任务成功率（Task Success Rate）

关注的是：

```
用户的问题是否真正被解决
```

例如：

用户问：

> 怎么申请退款？
> 

如果 AI：

- 给出正确步骤
- 或成功创建退款流程

才算成功。

---

### 用户满意度（CSAT）

关注的是：

```
用户主观感受
```

例如：

用户可能因为：

- AI语气礼貌
- 回答看起来很专业

而给出好评。

但问题可能 **没有真正解决**。

---

## 2 为什么会出现这种情况

客服场景中常见原因包括：

### ① 用户判断错误

用户可能：

- 没意识到答案是错的
- 没意识到政策限制

---

### ② 情绪被安抚

AI可能：

- 很礼貌
- 表达同理心

用户情绪被安抚，因此给好评。

---

### ③ 用户没有继续追问

有些用户：

```
放弃继续提问
```

但仍然给了好评。

---

## 3 评测体系应该怎么处理

因此在评测体系中通常会：

**把两个指标分开监控。**

例如：

| 指标 | 含义 |
| --- | --- |
| Task Success Rate | 任务是否完成 |
| CSAT | 用户满意度 |
| Follow-up rate | 用户是否继续追问 |

如果出现：

```
CSAT ↑
Task Success Rate ↓
```

说明模型可能只是：

```
“看起来很好”
```

但没有真正解决问题。

---

# 总结

> 如果用户的问题没有真正解决，但用户仍然给了好评，我通常不会把它算作任务成功，因为任务成功率和用户满意度是两个不同维度。任务成功率关注的是问题是否真正被解决，而用户满意度反映的是主观体验。在评测体系中通常需要同时监控任务成功率、用户满意度以及追问率等指标，通过多维度指标判断系统的真实表现。
> 

# **如果任务成功率很高，但用户满意度却很低，你会怎么分析？**

这是客服 AI 评测里一个**非常典型的诊断问题**。

如果 **任务成功率（TSR）很高，但用户满意度（CSAT）很低**，通常说明 **系统功能上是对的，但用户体验存在问题**。我一般会从 **四个层面逐步排查：任务定义、回答方式、系统体验和用户预期**。

---

## 1 先验证任务成功率的定义是否合理

首先要确认：

```
任务成功率是否定义正确
```

有时 TSR 可能只检测：

- 工具调用成功
- 返回正确信息

但没有确认 **用户是否真正完成目标**。

例如：

用户问：

> 如何退款？
> 

AI回答：

> 请在订单页面点击退款申请。
> 

系统会判定：

```
任务成功
```

但用户可能：

- 找不到按钮
- 操作步骤不清晰

因此需要检查：

- TSR 是否过于宽松
- 是否真正反映用户目标

---

## 2 分析回答的可执行性（Actionability）

即使答案正确，也可能：

- 步骤不清晰
- 过于复杂
- 不够具体

例如：

```
正确答案 ≠ 好答案
```

客服场景中更重要的是：

```
是否给出可执行步骤
```

例如：

- 第一步点击哪里
- 第二步填写什么

---

## 3 检查用户体验问题

用户满意度低可能来自系统体验问题，例如：

| 问题 | 影响 |
| --- | --- |
| 回答过长 | 阅读负担大 |
| 语气生硬 | 体验差 |
| 延迟高 | 用户等待 |
| 多轮对话复杂 | 操作成本高 |

因此需要分析：

- latency
- conversation turns
- abandon rate

---

## 4 分析用户预期

用户满意度还可能和 **预期差距**有关。

例如：

用户希望：

```
AI直接帮我退款
```

但系统只能：

```
提供退款步骤
```

虽然任务 technically 完成，但用户仍然不满意。

---

# 总结

> 如果任务成功率很高但用户满意度很低，我通常会从三个方面分析：首先验证任务成功率的定义是否合理，是否真正反映用户目标；其次分析回答的可执行性和表达方式，例如是否提供清晰步骤；最后检查系统体验问题，例如延迟、对话轮数以及用户预期差异。通过结合用户行为数据和案例分析，可以定位满意度下降的具体原因。
> 

# **如果任务成功率和用户满意度都很高，但公司成本却在上升，你会怎么看？**

如果 **任务成功率（TSR）和用户满意度（CSAT）都很高，但公司成本却在上升**，我会把它当成一个 **“质量—效率—成本”平衡问题** 来分析。客服 AI 的目标不仅是解决问题，还要 **在合理成本下解决问题**。因此我通常会从 **三层分析：指标结构 → 成本来源 → 优化策略**。

---

## 1 先检查评测体系是否缺少“效率指标”

很多评测体系只关注：

- Task Success Rate
- User Satisfaction

但忽略了：

```
效率和成本
```

例如：

- Token 使用量
- 模型调用次数
- 延迟（latency）

因此需要在评测体系中加入：

| 指标 | 含义 |
| --- | --- |
| Cost per conversation | 每次会话成本 |
| Average token usage | 平均 token 使用 |
| Latency | 响应时间 |

这样可以形成：

```
质量 + 成本 + 效率
```

三维评测。

---

## 2 分析成本上升的原因

在客服 AI 场景中，成本上升通常来自三个方面。

### （1）模型成本

例如：

- 使用更大的模型
- 推理成本增加

例如：

```
gpt-4 → gpt-4o
```

成本可能明显增加。

---

### （2）对话轮数增加

如果系统回答：

- 太复杂
- 太冗长

用户可能会：

```
继续追问
```

导致：

- conversation turns 增加
- token 成本增加。

---

### （3）工具调用成本

客服系统通常会调用：

- 订单系统
- 物流系统
- 工单系统

如果调用策略不合理，也会增加成本。

---

## 3 优化策略（保持质量同时降低成本）

通常可以通过三种方式优化。

### （1）模型分层

使用：

```
小模型 → 简单问题
大模型 → 复杂问题
```

降低整体成本。

---

### （2）优化 prompt 和回答长度

减少：

- 不必要解释
- 过长回答

提高信息密度。

---

### （3）优化任务流程

例如：

- 减少不必要的工具调用
- 优化 RAG 检索

---

## 总结

> 如果任务成功率和用户满意度都很高，但成本在上升，我通常会把问题放在“质量—效率—成本”的平衡上来分析。首先检查评测体系是否缺少效率指标，例如 token 使用量和对话轮数；其次分析成本来源，例如模型推理成本、对话轮数增加或工具调用策略；最后通过模型分层、prompt优化以及流程优化等方式，在保持质量的同时降低整体成本。
> 

# **如果降低成本会影响用户体验，你会怎么做？**

如果 **降低成本可能影响用户体验**，我一般不会简单做“降成本”或“保体验”的二选一，而是把问题当成 **质量—成本—效率的多目标优化问题** 来处理。核心思路是：**在用户体验的可接受范围内优化成本，而不是牺牲关键体验指标**。

---

## 1 先明确不可下降的核心指标（Guardrail Metrics）

首先需要确定哪些指标是 **不能明显下降的**，例如：

- Task Success Rate（任务成功率）
- CSAT（用户满意度）
- Safety（安全性）

这些指标可以作为：

```
guardrail metrics
```

如果降低成本导致这些指标明显下降，就不应该上线。

---

## 2 找到成本与体验的 Pareto 平衡点

通常会通过 **A/B test** 找到：

```
体验 vs 成本
```

之间的平衡点。

例如：

| 方案 | 任务成功率 | 成本 |
| --- | --- | --- |
| A | 92% | $0.20 |
| B | 91% | $0.12 |

如果：

- 成功率只下降 1%
- 成本下降 40%

可能是一个合理 trade-off。

---

## 3 做分层策略（Routing Strategy）

客服 AI 中常用的方法是：

**问题分级处理**

例如：

```
简单问题 → 小模型
复杂问题 → 大模型
```

这样可以：

- 保证复杂问题体验
- 降低整体成本

---

## 4 优化系统效率，而不是只换模型

很多成本问题其实来自系统设计，例如：

- RAG 检索过多
- prompt过长
- 不必要的工具调用

通过优化这些流程，可以：

```
降低成本但不影响体验
```

---

## 总结

> 如果降低成本可能影响用户体验，我通常会把问题当作多目标优化问题来处理。首先确定不可下降的核心指标，例如任务成功率和用户满意度作为 guardrail metrics；其次通过 A/B test 找到体验与成本之间的平衡点；同时通过问题分级 routing、系统效率优化等方式降低成本，而不是简单降低模型能力。
> 

# **如果不同用户对体验的要求不同，你怎么设计评测指标？**

在客服 AI 场景里，不同用户对体验的要求确实不一样，例如：

- 有些用户希望 **回答简洁、快速解决问题**
- 有些用户希望 **解释详细、步骤清晰**
- 有些用户情绪激动，更看重 **共情和安抚**

因此评测指标不能只用一个统一标准，而需要 **分层和分群设计评测体系**。我通常会从 **三个层面设计指标：统一核心指标、用户分群指标、个性化体验指标**。

---

## 1 统一核心指标（Core Metrics）

首先定义一组 **所有用户都适用的核心指标**，确保系统基本能力达标，例如：

- **Task Success Rate（任务成功率）**
- **Correctness（回答正确性）**
- **Safety（安全合规）**

这些指标是系统的 **基础能力要求**，无论用户类型如何都必须满足。

---

## 2 用户分群评测（Segment-based Evaluation）

然后根据用户类型进行 **分群评测（user segmentation）**。

例如客服场景可以按：

| 用户类型 | 评测重点 |
| --- | --- |
| 快速型用户 | 响应时间、回答简洁度 |
| 信息型用户 | 解释完整性 |
| 情绪型用户 | 共情和语气 |

这样可以避免：

```
同一个指标评价所有用户体验
```

的问题。

---

## 3 个性化体验指标（Experience Metrics）

在用户体验层面，可以设计一些更细粒度的指标，例如：

- **Response brevity（回答简洁度）**
- **Actionability（可执行性）**
- **Empathy score（共情程度）**

不同用户群体可以有不同权重。

例如：

| 用户类型 | 权重 |
| --- | --- |
| 快速型用户 | 简洁度权重高 |
| 复杂咨询用户 | 完整性权重高 |

---

## 4 结合用户行为数据验证

为了验证这些指标是否合理，还需要结合 **真实用户行为数据**，例如：

- follow-up rate（追问率）
- conversation turns（对话轮数）
- task completion rate（任务完成率）

通过这些数据验证评测指标是否真正反映用户体验。

---

## 总结

> 如果不同用户对体验要求不同，我通常会采用分层评测体系。首先定义统一的核心指标，例如任务成功率和安全性；然后根据用户类型进行分群评测，例如快速型用户更关注响应速度和简洁度，而复杂咨询用户更关注回答完整性；最后通过用户行为数据验证这些指标是否与真实用户体验一致，从而构建更贴近真实场景的评测体系。
>
